**Date:** 2026.05.14
**Topic:** Free tooling and AI co-design for SystemVerilog code coverage and formal assertion generation

## 1. Free Tools for SystemVerilog Code Coverage

### Open-Source Simulators

**Verilator**
- Most capable free SV simulator; supports line, toggle, and functional coverage
- Outputs `.dat` coverage files; includes `verilator_coverage` CLI for merging and annotating results
- Best for synthesizable RTL; some SV constructs have limited support

**Icarus Verilog (iverilog)**
- Basic Verilog/SV simulation; coverage support is limited — not recommended as a primary coverage tool

### Coverage Reporting & Visualisation

| Tool | Role |
|------|------|
| `verilator_coverage` | Ships with Verilator; merges `.dat` files and annotates source |
| LCOV / genhtml | Produces HTML reports with line-level highlighting (adaptable from C/C++ flows) |

### Formal & Structural Analysis

**SymbiYosys (sby)**
- Open-source formal verification frontend
- Supports `assert`, `assume`, `cover` SVA properties
- Works with Yosys; backends include SMTBMC, Boolector, Z3, ABC PDR

**Yosys**
- Synthesis tool; can instrument designs for toggle/expression coverage analysis

### Testbench Frameworks

- **CocoTB** — Python-based testbench framework; drives coverage collection when paired with Verilator
- **VUnit / UVVM** — Open-source verification frameworks; VUnit has coverage integration hooks

### Recommended Free Coverage Stack

```
Verilator (simulation + coverage instrumentation)
    → verilator_coverage (merge / annotate .dat files)
    → LCOV / genhtml (HTML visualisation)
```

Provides **line, toggle, and functional coverage** with readable HTML reports at zero cost.

> **Note:** Full `covergroup` / `coverpoint` / cross-coverage is best supported in commercial
> tools (Synopsys VCS, Cadence Xcelium, Mentor Questa). Verilator support is improving but
> not yet complete.

## 2. AI Co-Design of Formal Assertions

### Current Capability

AI-assisted SVA generation is an active and maturing field. LLMs can generate SystemVerilog
Assertions from RTL code and/or natural language specifications, significantly reducing manual
effort. Studies have shown LLM-generated assertions match human-written ones in readability,
with only marginal differences in correctness — making them a strong accelerator for early
assertion drafts.

### Key Approaches (by complexity)

| Approach | Description | Maturity |
|----------|-------------|---------|
| Direct LLM prompting | Feed RTL + spec to an LLM; ask for SVAs | Production-usable with review |
| Fine-tuned LLMs + iterative refinement | Custom compiler generates error messages fed back to LLM; 7.3× improvement in correct assertions vs baseline | Research / early adoption |
| Knowledge Graph + RTL fusion (AssertionForge, Nvidia) | Builds a KG from spec + RTL to capture internal signal interactions missed by spec-only methods | Research |
| MCTS-guided generation (SANGAM) | LLM-guided Monte Carlo Tree Search; 204% more correct assertions vs prior LLM methods | Research |
| Commercial: iSpec.ai (Agnisys) | Natural English → SVA service, targets both simulation and formal verification | Available now |

### SymbiYosys Compatibility of AI-Generated SVAs

SVAs generated by LLMs are compatible with SymbiYosys **provided they use synthesizable constructs only**.

| SVA Feature | SymbiYosys Support |
|---|---|
| `assert property (...)` | ✅ Full |
| `assume property (...)` | ✅ Full |
| `cover property (...)` | ✅ Full |
| Sequence operators (`##`, `[*]`, `\|->`, `\|=>`) | ✅ Mostly supported |
| `$past()`, `$stable()`, `$rose()`, `$fell()` | ✅ Supported |
| Explicit clocking `@(posedge clk)` | ✅ Required |
| `disable iff` | ⚠️ Partial support |
| `covergroup` / `coverpoint` / cross-coverage | ❌ Not supported (simulation only) |

### Common LLM Failure Modes to Watch For

- Off-by-one clock cycle errors (most common)
- Referencing signal names not present in the RTL
- Vacuous assertions (trivially true antecedent — always run `cover` mode to check)
- Redundant assertions that miss some conditional branches in complex expressions
- Mixing assignment checks with preconditions

### Recommended AI → Formal Verification Workflow

```
1. Prompt LLM with RTL + spec
   └─ Specify SymbiYosys-compatible SVA only (see prompt template)

2. Review generated assertions
   └─ Check signal names, clock edges, cycle offsets

3. Run SymbiYosys in prove / cover mode
   └─ symbiyosys -t prove design.sby

4. Feed failures / vacuity warnings back to LLM (refinement prompt)
   └─ Iterate 2–3 rounds until convergence
```

## 3. Toolchain Summary

| Phase | Free Tool | Notes |
|-------|-----------|-------|
| RTL Simulation | Verilator | Best SV support in open-source |
| Coverage Collection | verilator_coverage | Line, toggle, functional |
| Coverage Reporting | LCOV / genhtml | HTML output |
| Formal Verification | SymbiYosys + Yosys | assert / assume / cover |
| Formal Backends | SMTBMC, Boolector, Z3, ABC | Included with sby |
| Testbench | CocoTB / VUnit | Python-driven, integrates with Verilator |
| Assertion Generation | LLM (Claude, GPT-4, etc.) | With review + iteration |
