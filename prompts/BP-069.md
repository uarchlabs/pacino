## Background
bp_history_decisions.md and bp_history_interfaces.md are the
authority for this task. bp_history.sv is the file to modify. The
two package files are read-only reference for parameter and struct
names; do not modify them. If a required parameter is genuinely
absent from the packages, stop and report it rather than adding it.

The current bp_history.sv (2026-05-21) is caller-owned-pointer:
ghist_ptr / phist_ptr are inputs, and rollback is driven by an
external pointer pair (rollback_ghist_ptr / rollback_phist_ptr).

The planning decision (bp_history_decisions.md section 2) rules the
pointer module-owned: the live pointer is an internal register the
module advances itself, exposed as a registered output; rollback
supplies a checkpoint INDEX, and the module restores the pointer
from its own checkpoint array. This task makes the RTL match the
target interface.

The dual-slot update, rollback-wins priority, and stale-fold-on-
rollback behavior already exist in the RTL and are correct; they
are retained. The change in this task is the pointer ownership and
the rollback port.

## Binding Previous Decisions
- Pointer is module-owned (decisions section 2). ghist_ptr /
  phist_ptr are registered OUTPUTS, advanced inside the module by
  num_branches each cycle (0, 1, or 2), modulo buffer width.
  Sequential-only advance: reset and checkpoint-restore are the
  only non-incremental loads. No external pointer-load path.
- Rollback is by index (decisions section 2.2). Replace the
  rollback_ghist_ptr / rollback_phist_ptr inputs with a single
  rollback_ckpt_idx input (FTQ_IDX_BITS wide). On rollback_valid,
  read the internal checkpoint at that index, load the live
  pointer from it, and recompute all folds from the restored
  pointer position (the existing fold_ghr recompute, now keyed off
  the checkpoint-read pointer instead of the external port).
- Rollback wins (decisions section 4): rollback_valid and
  num_branches > 0 in the same cycle -> rollback executes, the
  prediction update and checkpoint write are dropped. Keep the
  existing mutually-exclusive if/else structure.
- Stale folds on rollback (decisions section 5): the fold
  registers are not cleared on rollback; existing behavior, no
  change.
- Dual-slot fold order is slot 0 then slot 1 (decisions section
  3.3). Existing nested fold_step, no change.
- num_branches semantics (decisions section 3.1): 0 holds all
  state including the pointer; 1 advances by 1; 2 advances by 2.
  Value 3 undefined.
- Checkpoint stores pointer pair only, written on ckpt_wr_en at
  ckpt_wr_idx, exposed on ckpt_ghist_ptr / ckpt_phist_ptr
  (interfaces, Checkpoint Interface). Existing checkpoint storage
  is retained; it is now also the rollback read source.

## Assumptions made not explicit in the prompt
- Model/effort reported as claude-opus-4-8 normal.
- Pointer advance is placed unconditionally in the normal-update
  branch as ptr + num_branches (mod width). num_branches=0 yields
  +0, which holds the pointer, satisfying the "0 holds all state"
  requirement without a separate guard.
- Checkpoint write stores the CURRENT (pre-advance) pointer value.
  Under nonblocking semantics, ckpt_*_ptr <= ghist_ptr_r captures
  the pointer value before this cycle's advance. This matches
  decisions section 6 ("writes the current internal pointer pair")
  and requirement 5 ("preserve the checkpoint write path"). See the
  discrepancy note below.

## Decisions made not explicit in the prompt
- The rollback_valid branch was given a named block (rb_apply) with
  block-local temps rb_gptr / rb_pptr (blocking-assigned from the
  checkpoint arrays, then used for the pointer load and every
  fold_ghr recompute). This keeps the recompute lines within the
  80-column limit and reads the checkpoint once per rollback rather
  than re-indexing the array in each of the ~22 fold expressions.
  Registers are still nonblocking-assigned; only the local temps
  use blocking assignment.
- The pointer-advance modulo result (int) is width-cast with
  GHIST_PTR_BITS' / PHIST_PTR_BITS' to avoid a Verilator WIDTHTRUNC
  warning on assignment to the pointer registers.
