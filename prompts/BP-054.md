## Hypothesis

Given seeded table entries, the ITTAGE prediction path selects
the correct provider and alternate, sets using_primary per the
null-CTR and UAON rule, sets pred_strong from the selected
provider's CTR, drives the selected provider's target, and these
outputs are valid at s2. The prediction path has only been
exercised as setup for update tests, so its correctness as the
unit under test is assumed, not established.

## Background

Prediction is the read path. Across the tables that hit on tag,
the primary is the longest-history hit and the alternate is the
next-longest hit. Selection: primary is used when its CTR is not
null (3'b000); on null primary CTR, the alternate is used when
uaon >= IT_UAON_THRES, else the primary. using_primary reflects
the result of that selection. pred_strong reflects CTR-not-null
on whichever component actually supplied the prediction. The
predicted target is the selected provider's TGT field.

Per the cluster pipeline, ITTAGE produces its prediction at s2
(alongside FTB and TAGE). TD #42 notes a diagram error placing
it at s3; this task confirms the outputs are valid at s2 in RTL.

The authority is ittage_cntrl_decisions.md.

Component index 0 is the no-component sentinel. The UAON counter
update path is proven (#59); this task uses UAON only as a read
input to the selection mux, with the counter forced to a chosen
value, not trained.

## Binding Previous Decisions

- Component index 0 is the no-component sentinel for prm_comp,

  Do not train CTR or UAON to reach a state; force it.
