# Task Overview

BP-049 verified that the target field is written when the
provider CTR is zero on misprediction, but exposed a defect it
did not fix: the per-table target write enable gates on
(prm_match | alt_match), so when a target write occurs both the
provider and the non-provider hitting table have their target
field overwritten. The specification requires that only the
provider's target field is modified. BP-049 did not detect this
because ittage_interfaces.md, which states the gating rule, was
not in its context, and its tests read back only the provider
entry. This task corrects the target write gating so only the
provider table is written, in both provider regimes, and extends
the tests to read back the non-provider entry and confirm it is
unchanged.

## Hypothesis

The target write enable in ittage_table.sv gates on
(prm_match | alt_match). When a target write occurs on
misprediction with a zero provider CTR, this writes the target
field of both the provider and the non-provider hitting table.
The specification requires only the provider's target be written.
Gating the target write on the provider table selector match
alone -- prm_tbl_sel for UP=1, alt_tbl_sel for UP=0 -- corrects
the behavior. The non-provider target then remains unchanged in
both provider regimes, confirmed by readback.

## Background

ittage_interfaces.md, section "Target Write Gating (tgt_wr_u0)",
states the rule: the target field is written only when
indir_mispredict, the provider CTR is null, AND THIS_TABLE
matches the provider table selector (prm_tbl_sel_u0 when
ittage_using_primary, alt_tbl_sel_u0 otherwise). The section
states explicitly that only the provider's target field is
updated, even when the conditions hold for both hitting tables.

BP-049 found the RTL gated the target write on (prm_match |
alt_match) rather than the provider selector match. It treated
the resulting non-provider write as permitted because the
interfaces file was not in its context. The non-provider here is
the primary (longer-history, higher-priority) table when the
alternate is the provider; corrupting its target affects the
next prediction at that PC, where the primary is selected first.

BP-049 added TC-TGT-A through D. TC-TGT-D reads back only the
provider (alternate) target and does not check the non-provider
(primary) target, so it passed despite the defect.

The backdoor RAM-seeding method (BP-044b) is the test mechanism.
Entry packing: VAL[0], CTR[3:1], USE[5:4], EPC[7:6], TGT[45:8],
TAG[ALLOC_DATA_WIDTH-1:46]. Address split: bank =
idx[INDEX_BITS-1], ent = idx[INDEX_BITS-2:0]. bw_write writes a
full entry to u_ram_s0/u_ram_s1 in the generated table.

## Assumptions made not explicit in the prompt

- The split-strobe approach (t_prm_tgt_wr_u0 / t_alt_tgt_wr_u0) was
  chosen over a single strobe + using_primary flag passed into
  ittage_table. A prior attempt using using_primary_u0 as a table
  input produced incorrect simulation results due to Verilator
  scheduling ambiguity with partial-bit generate-loop assignments.
  The split-strobe approach avoids this by using the same per-slot
  generate loop pattern already proven correct for CTR strobes.
- TC-TGT-B-ext uses direct RAM backdoor reads rather than a
  prediction-based readback for the post-update check. A prediction-
  based readback is ambiguous because the mispredict triggers an alloc
  write to IT3, which then appears as the primary on the next
  prediction at the same PC, displacing IT2 and IT1. The direct read
  unambiguously confirms IT1's TGT is unchanged.

## Decisions made not explicit in the prompt

- UAON-decrement step retained in TC-TGT-B-ext: UAON resets to
  IT_UAON_THRES=8 which would trigger use_alt if prm_ctr=0 and alt is
  present. A correct prediction at PC=0x1000 with IT1-only hit
  decrements UAON 8->7 before the main test, ensuring using_primary=1
  with both IT2 (prm, CTR=0) and IT1 (alt, CTR=1) present.
