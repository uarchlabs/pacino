## Hypothesis

The ITTAGE EPC write enable is gated provider-only and writes
the correct EPC value to exactly the provider table entry.
The competing hypothesis is that epc_we carries the same
(prm_match | alt_match) gate that affected CTR, USE, and TGT,
so an EPC-writing update also corrupts the non-provider table
entry. A readback that inspects only the provider cannot tell
these apart -- that gap is exactly what let BP-049 miss the
TGT defect (#57). This task settles the question by readback
of both the provider and the non-provider entry.

## Background

EPC is a control field (included in IT_CNTRL_BITS_WIDTH) and
is written on update as a USE rider. The write enable signals
are epc_we_s0/s1 in ittage_cntrl, consumed by the EPC write
gate in ittage_table. BP-044c changed these enables alongside
use_we; no positive readback test was ever added.

The provider-gating defect class was found and fixed three
times: CTR (BP-044b), USE (BP-044c), TGT (BP-049a). In each
case the per-table write gate used (prm_match | alt_match),
writing both hitting tables instead of the provider only.
BP-049a corrected TGT by splitting the single write strobe
into per-slot provider strobes (t_prm_tgt_wr_u0 for UP=1 /
prm_ctr==0, t_alt_tgt_wr_u0 for UP=0 / alt_ctr==0), with the
table gate then (prm_tgt_wr & prm_match | alt_tgt_wr &
alt_match) -- mutually exclusive, provider table only.

BP-049a also recorded a Verilator v5.048 trap: a first attempt
that drove the table write from a using_primary table-input
selecting partial bits in a generate assignment was scheduled
incorrectly and produced wrong results. The split-strobe
approach reusing the proven per-slot CTR/USE strobe pattern is
the one that worked.

ittage_interfaces.md "Target Write Gating" was made explicit
in session-047 that only the provider field is written even
when both hitting tables satisfy the conditions. The EPC write
is governed by the same provider-only invariant.

## Binding Previous Decisions

- Provider-only write is the binding invariant. Only the
  provider table entry's EPC field is written on an
  EPC-writing update, even when both hitting tables would
  otherwise satisfy the write condition.
- UP=1 means the primary table is the provider; UP=0 means the
  alternate table is the provider. The non-provider in each
  case is the other hitting table.
- Readback through prediction is preferred. Where post-alloc
  prediction ambiguity applies, use a direct RAM read instead
  (the resolution adopted in BP-049a). Seed entries via the
  bw_write backdoor.
- If an RTL fix is required, use the per-slot provider
  split-strobe pattern proven for TGT in BP-049a. Do not drive
  the EPC write from a using_primary table-input with
  partial-bit generate assignments -- that pattern was scheduled
  incorrectly by Verilator v5.048 in BP-049a.
