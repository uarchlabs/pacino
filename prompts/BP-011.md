This is a genuine RTL defect. The index and tag fields in captured prediction meta are always zero. In hardware this means every update driven from FTQ-captured meta would write to address 0 regardless of the branch PC. The round-trip tests worked only because the testbench overrides the captured meta with PC-computed values. This needs a fix session.
Discrepancy 2: T0 prm_ctr mis-extracted
This is also a genuine RTL defect. tage_cntrl extracts T0 CTR from the wrong bit positions of cntrl_bits. The existing update tests masked this because they build meta manually. Needs a fix session.
Your assessment question about T0 CTR padding:
You asked whether the workaround is picking the correct MSB position. With the workaround overriding meta1.tage_prm_ctr = 3'b010, the update logic sees CTR=3'b010 and increments to 3'b011. So the direction interpretation is correct because tage_prm_pred_tkn was captured correctly at predict time from t_taken_p1 (the actual RAM MSB), not from tage_prm_ctr[2]. The workaround is sound for the test but the root cause still needs fixing.
