## Claude.ai Assessment

Your assessment is correct. The core problem is clear: when TAGE_FAST_INIT=1, sram_init still runs its full 2048-cycle sequence before asserting tbl_ri_rdy, so tage_rdy takes the same 2049 cycles regardless. The fast init path is functionally a no-op from the testbench perspective.
Root cause: tage_rdy is hardwired to tbl_ri_rdy from sram_init. sram_init has no knowledge of TAGE_FAST_INIT and always sequences through all entries. The RAM contents are correct at t=0 when FAST_INIT=1, but tage_rdy still waits for sram_init to finish.
The fix: tage.sv needs to read +TAGE_FAST_INIT and drive tage_rdy directly. The logic is:
systemverilogassign tage_rdy = fast_init ? 1'b1 : tbl_ri_rdy;
Where fast_init is read via $value$plusargs in an initial block in tage.sv and held in a logic register. sram_init still runs — its tbl_ri_active will fire and write over the fast-initialized values — so tbl_ri_active also needs to be suppressed when fast_init=1.
This means tage.sv needs:

Read +TAGE_FAST_INIT into a logic fast_init_r
Gate tbl_ri_active from sram_init: assign tbl_ri_active_gated = fast_init_r ? 1'b0 : tbl_ri_active_raw
assign tage_rdy = fast_init_r ? 1'b1 : tbl_ri_rdy

Also noted: tage_tb_decisions.md needs correcting — the "few cycles" language was mine and was wrong. It should be updated before BP-010b.
Two questions before I write BP-010b:

Should sram_init be fully bypassed when fast_init=1 (gate tbl_ri_active to zero as above), or should it be left running harmlessly while tage_rdy asserts immediately?

## Background

Read tage_tb_decisions.md in full before proceeding.
It is the authoritative source for all design decisions
in this session. Key points are summarized below.

TAGE_FAST_INIT is a runtime plusarg (+TAGE_FAST_INIT=1).
It is read inside initial blocks in tage_bim.sv and
tage_table.sv only. bw_ram.sv is NOT modified.
The initial blocks write the bw_ram mem arrays via
hierarchical reference to bypass the sram_init sequence.

When TAGE_FAST_INIT=0 (default): sram_init sequences
through all entries. tage_rdy asserts after approximately
(1 << MAX_IDX_WIDTH) cycles from reset deassertion.

When TAGE_FAST_INIT=1: initial blocks write all bw_ram
mem entries at time zero. sram_init completes within a
few cycles. tage_rdy asserts shortly after reset
deasserts.

TAGE_SRAM_INIT_VALUE is a new parameter added manually
to bp_defines_pkg.sv. It is the value written to each
RAM entry by both sram_init and the initial blocks.
tage_bim casts to THIS_CTR_WIDTH. tage_table casts to
ALLOC_DATA_WIDTH.

tage_rdy is a new output port on tage.sv. It is driven
by: assign tage_rdy = tbl_ri_rdy. tbl_ri_rdy is the
ready output of sram_init already present in tage.sv.

The magic number used in tage.sv for sram_init .INIT_VAL
must be replaced by TAGE_SRAM_INIT_VALUE.

## Assumptions made not explicit in the prompt
A1. bw_ram mem array is 2D: logic [WIDTH-1:0] mem [BANKS][ENTRIES]
    where BANKS=2. The tage_tb_decisions.md initial block pattern
    shows mem[i] (1D access) looping over RAM_ENTRIES, which would
    be out of bounds for the BANKS dimension. Implemented correct
    2D access: mem[b][i] with an outer bank loop b=0..1.
    This affects both the initial blocks in tage_bim/tage_table and
    the hierarchical reference checks in tb_tage.sv.

A2. Verilator requires constant indices for hierarchical references
    into generate block instances. The loop for t=1..4 in the
    tage_table RAM check was unrolled to use explicit constant
    paths: gen_tage_tbl[1], gen_tage_tbl[2], gen_tage_tbl[3],
    gen_tage_tbl[4].
