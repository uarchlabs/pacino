## Objective

Fix the structural gap identified in BP-004c Deferred Work #2.
The loop predictor cannot learn iteration counts from a cold start
because all update conditions gate on pred_is_loop=1, which requires
conf==LP_CONF_LEVEL, which can never be reached from conf=0 without
an update. The fix adds a learning update path that increments
curr_itr on any actual_taken hit regardless of confidence, gating
only the prediction output on confidence.

### 2. loop_pred.sv

Two changes:

**Prediction path:**
In the s0 combinational block, set pred_comb.lp_hit=1
whenever any_hit is true, independent of confidence.
pred_comb.pred_is_loop remains gated on confidence as
before.

**Update path:**
Add a new learning update condition in the always_comb
update block, evaluated after the existing four conditions:

  Condition 5 (learning):
    upd_p0.lp_hit && !upd_p0.pred_is_loop && upd_p0.actual_taken
    -> increment curr_itr, no confidence change, no
       pred_is_loop assertion, no allocation.

Condition 5 must not overlap with condition 1
(actual_taken && pred_is_loop). Gate condition 5 on
!upd_p0.pred_is_loop explicitly. Priority: conditions
1-4 take precedence; condition 5 is the fallthrough
learning path.

### Assumptions made not explicit in the prompt
- pred_comb.lp_hit is not gated by pred_valid_p0. The task
  stated "independent of confidence" but did not address
  pred_valid_p0. lp_hit reflects the raw tag match result; the
  FTQ/update path gates writes with upd_valid_p0, so a spurious
  lp_hit when pred_valid_p0=0 does not cause a false write.
- The ff block write condition was extended to pred_is_loop ||
  lp_hit (not just lp_hit && actual_taken). The task did not
  specify the ff block change but it is required: without it,
  condition 5 writes upd_entry_hit but the ff block never
  commits it, and conditions 2/3 can never advance cnf for
  non-trusted entries. This is the mechanism behind the
  expected outcome "advance conf from 0 toward LP_CONF_LEVEL".
- Condition 5 placed between conditions 1 and 2 (not after
  condition 3). Placement after condition 1 ensures
  actual_taken=1 with pred_is_loop=0 is caught by condition 5
  before condition 2 can misfire (condition 2 does not check
  actual_taken explicitly). This satisfies the "conditions 1-4
  take precedence" requirement while keeping condition 5
  reachable.

### Decisions made not explicit in the prompt
- Conditions 4 and 1 required explicit pred_is_loop guards to
  prevent them from consuming condition 5's case. Without the
  guard on condition 4, any lp_hit && !pred_is_loop &&
  actual_taken event (pred_taken=0 for all non-trusted entries)
  would have been caught by condition 4 (which resets cnf and
  curr_itr), making condition 5 dead code. The prompt implied
  this by stating "conditions 1-4 take precedence" and requiring
  condition 5 to be reachable.
- The confidence advancement path (conditions 2/3 writing for
  lp_hit entries) was inferred from the expected outcome. The
  full learning loop requires: condition 5 on taken iterations,
  condition 3 on wrong exits (stabilizes past_itr), condition 2
  on correct exits (increments cnf). All three are now active
  for non-trusted entries via the ff block change.
