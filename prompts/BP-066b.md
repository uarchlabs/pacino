## Background
The position field (FTB_BR_POS_BITS = 3 bits per conditional and for
the jump) is the in-block instruction slot (0..7) a branch occupies. It
has always been stored in the entry but had no producer and no consumer
(BP-066 wrote it as 0). FTB-4 resolves this by adding the ports.
Authority: ftb_decisions.md (4 entry format, 5.4/5.5),
ftb_interfaces.md (2.3 outputs, 2.5 input, IC-FTB-15). The field already
exists in the 105-bit RAM entry, so there is NO width change and NO
package edit.

## Binding Previous Decisions
- Producer: add input ftb_upd_pos_u0 [FTB_BR_POS_BITS-1:0] to the update
  port. It is the resolving branch's in-block slot, supplied by the
  FTQ/resolve side. ftb_cntrl writes it to the position of the field
  being filled -- the conditional selected by ftb_upd_br_idx_u0 (when
  ftb_upd_is_br_u0) or the jump field (when ftb_upd_is_jmp_u0).
- Position is static per filled field. Write it ONLY when the field is
  first filled (allocate or free-field fill, ftb_decisions.md 5.4). Do
  NOT rewrite it on an in-place conf/target update (5.5) -- the branch
  does not move. It is reset only by reallocation (overwritten when the
  field is filled with a new branch).
- Consumers: add outputs ftb_br0_pos_p2 / ftb_br1_pos_p2 /
  ftb_jmp_pos_p2 [FTB_BR_POS_BITS-1:0], the stored position of each
  field, pipelined to p2 alongside the other prediction outputs.
- No width change. FTB_BR_POS_BITS already exists and the position bits
  already occupy the entry (the RAM entry typedef already has the pos
  subfields; BP-066 simply fed them 0). FTB_RAM_ENTRY_WIDTH stays 105.
  No package edit; do not modify bp_defines_pkg.sv beyond confirming
  FTB_BR_POS_BITS, and do not modify bp_structs_pkg.sv.
- ftb top (ftb.sv): pass through the new ftb_upd_pos_u0 input and the
  three pos outputs on the boundary and the ftb_cntrl instantiation.
  Structural only.
- ftb_array.sv and ftb_plru.sv are NOT touched (no width change, the
  position field is part of the existing ftb_array data entry written by
  ftb_cntrl).

  jump uses the analogous jmp_fresh against upd_old.jmp.valid. This
  reuses the existing first-fill test rather than introducing a new
  one, keeping the position-static rule (5.5) consistent with the
  existing conf init rule (5.4).
- The three pos outputs are driven as ungated data outputs (no
  flush_clr gating), matching the existing treatment of conf and
  target. The downstream consumer qualifies them with the gated
  valids, exactly as for conf / target.

- The jump fresh test reads upd_old.jmp.valid. On a hit, upd_new is
  seeded from upd_old (so jmp.pos already equals the stored slot); the
  explicit else-branch (upd_old.jmp.pos) makes the static-on-rewrite
  behavior self-evident at the assignment site rather than relying on
  the seed.
