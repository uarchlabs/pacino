   Diagnosis:
   The USE write path (HAND-FIX-001) used cascaded
   assign statements:
     assign prm_match_s0 = (prm_tbl_sel_u0[0] == ...);
     assign prm_alt_match_s0 = prm_match_s0 | alt_...;
     assign use_we_s0 = ... & prm_alt_match_s0;
   A prior fix (BP-019a investigation) removed the
   prm_alt_match_s0 intermediate and inlined the OR:
     assign use_we_s0 = ... & (prm_match_s0 | ...);
   The inline fix was insufficient. Debugging with an
   always_ff posedge-capture block in the testbench
   confirmed: at the USE write clock edge, prm_match_s0
   evaluated to 1 (correct) but use_we_s0 evaluated to
   0 (wrong). The simulator evaluated use_we_s0 before
   prm_match_s0 in its dependency traversal, producing
   a stale 0 for prm_match_s0 when computing use_we_s0.
   This is a simulator evaluation-order ambiguity with
   cascaded continuous assign statements.

   Fix applied (same change for s0 and s1):
   Replaced all individual assign statements that
   compute match signals and write-enables with a
   single always_comb block per slot. Within a single
   always_comb block, statements execute sequentially,
   so prm_match_s0 is computed before use_we_s0 reads
   it. This eliminates the ordering ambiguity.

   Before (s0 example, same pattern for s1):

     assign alt_match_s0 = (alt_tbl_sel_u0[0] == ...);

## Other Notes

Root-cause detail: the Verilator 5.020 simulator
evaluates continuous assign statements using an
internal dependency DAG. Under certain conditions,
when multiple assign statements form a chain (A->B->C),
the simulator can evaluate C before A completes its
update, reading a stale value of B. This is not a
bug in the RTL logic itself -- the behavior is correct
at synthesis. It is a simulation-model evaluation
ordering issue specific to the cascaded assign pattern
combined with how Verilator schedules evaluation of
changed signals.

The fix (always_comb) is both correct for synthesis and
unambiguous for simulation: always_comb blocks are
evaluated in textual order within the block, so the
dependency chain is resolved top-to-bottom.

This issue affected BOTH slots (we_s0 and we_s1) since
both had the same cascaded assign pattern. Both were
fixed in this session.
