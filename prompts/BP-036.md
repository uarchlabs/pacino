## Assumptions made not explicit in the prompt

A1. The prompt requires tbl_hit_p1 to be driven one
    cycle after pred_val_p0 (Binding Decision 7). In
    Verilator's coroutine model, "one cycle later" means
    the signal must already be present at the POSEDGE
    that latches p0->p1, not set after it. The pred_s0
    task and tc_pred_06 drive these signals before the
    posedge and clear them after. This is consistent with
    BD7 intent and required for correct simulation.

A2. The meta_p1 intermediate and g_meta block are RTL
    internal implementation details not visible in the
    module interface. Removing them is considered an RTL
    fix (same observable behavior, correct scheduling)
    rather than a behavioral change.

## Decisions made not explicit in the prompt

D1. prv_alt_scan gating: gated the entire prv_alt_scan
    body on pred_val_p1[s] with explicit 0 defaults
    outside the gate. This forces Verilator to classify
    the block as nba_sequent (it reads a FF output) rather
    than stl_sequent (evaluated once at simulation start).
    Without this, tbl_hit_p1 changes are invisible to
    prv_alt_scan.

D2. meta_p2_reg direct-read: eliminated the meta_p1
    wire layer so the always_ff reads prm_hit and
    prm_comp directly. Reduces dependency chain depth
    and avoids a second scheduling inversion where
    nba_sequent placed g_meta after the meta_p2_r
    DLY capture.

## Other Notes
Verilator 5.020 scheduling classification determines
whether an always_comb block is placed in stl_sequent
(cold, run once), act_comb (active, triggered by
posedge/negedge), or nba_sequent (NBA phase, re-run
after FF updates). The key rule: if all reads in a
block are of module INPUT ports, Verilator classifies
the block as stl. Adding a read of any FF output
(even gated by if) causes reclassification to
nba_sequent. This is the root cause of the prv_alt_scan
defect and the fix strategy.
