## RVA23 compliance risks and gaps noticed

SIMULATION NOTE: Prompt stated "No staging always_ff is required
for prediction stimulus -- tage_pred_inp_p0 is a struct input,
not a RAM write path." This is incorrect for Verilator 5.020
--timing. Blocking assignments to struct-typed array elements in
initial block coroutines do not set simulator activity flags.
always_comb blocks downstream of pred_inp_p0.pc (addr_mux_s0 in
tage_bim.sv) are not re-evaluated. Without staging, the bw_ram
registers address=0 at the prediction posedge, reading the
FAST_INIT default (0) instead of the pre-loaded value. The fix is
identical to the update path: use always_ff staging via NBA. This
is a Verilator simulation artifact; it does not affect synthesis
or RTL correctness. The feedback memory was already correct:
feedback_verilator_timing_struct.md confirmed only always_ff NBA
triggers the correct evaluation chain.

DISCREPANCY: tage_use_alt_on_na semantics differ between the
planning document (prompt spec) and RTL.

Prompt (TC-E spec) states: "Verify tage_use_alt_on_na==0" when
uaon counter is below threshold.

RTL (tage_cntrl.sv line ~537):
  meta_p1[s].tage_use_alt_on_na = uaon_trig_p1[s];

uaon_trig_p1 is set whenever prm_comp != 0 AND prm CTR is in
{011, 100}. It does NOT test whether the counter exceeds the
threshold. The UAON mux switch uses uaon[s][3], but the trigger
flag records only whether the trigger CONDITION was met, not
whether the mux acted on it.

Impact: The update path consumer checks use_alt_on_na when
deciding whether to update the UAON counter. If the flag
semantics are "trigger fired" rather than "alt was actually used",
the UAON counter update logic will apply its rules correctly per
tage_cntrl_uaon_update_rules.md. No functional error in the
update path is expected.

However, upstream consumers reading meta.tage_use_alt_on_na
as "alt was actually used" would misinterpret it. This is a
potential software-visible semantic gap.
