## Hypothesis
ras.sv can be implemented as a single self-contained register-
file-based module with no synchronous SRAMs, no PQ/UQ, and no
credit arbiter. All push/pop logic, the same-cycle slot0-to-
slot1 bypass, the recursion counter, and the p3 repair path are
expressible without inter-module dependencies beyond the package
imports.

## Background
The RAS maintains two independent register-file-based stacks:

Speculative stack: 16 entries, simple circular buffer.
Pointers TOSR, TOSW, BOS checkpointed into the FTQ at every
prediction for O(1) mispredict recovery. Recovery is pointer-
only; no data is cleared or replayed.

Commit stack: 32 entries, conventional circular stack.
Pointer CSP. Updated at retire. Provides fallback TOS value
when the speculative stack is empty.

Both stacks carry: ret_addr (VA_WIDTH bits), rctr (4b).

Pipeline stages:
  p0/s0: combinational TOS read, initial prediction.
  p2/s2: FTB-confirmed push/pop, two slots per cycle,
         slot 0 processed before slot 1.
  p3/s3: repair pass, inverse operation if p3 FTB
         prediction disagrees with p2 operation.

Note: planning documents use s-stage notation (s0/s1/s2/s3).
RTL port names use p-stage notation (p0/p1/p2/p3). They are
equivalent. Use p-stage in all RTL.

## Binding Previous Decisions
- RAS_SPEC_ENTRIES=16, RAS_COMMIT_ENTRIES=32, static partition.
  No unified pool. (ras_decisions.md section 3)
- RAS_RCTR_WIDTH=4. Recursion counter in scope for this task.
  (ras_decisions.md section 5)
- Simple circular buffer speculative stack. No linked list.
  (ras_decisions.md section 3.2)
- Branch type classification is FTB's responsibility. RAS
  receives bp_br_type_e per slot and acts on DIRECT_CALL,
  INDIRECT_CALL (push) and RETURN (pop). RAS does not
  classify instructions independently.
  (ras_interfaces.md IC-RAS-01, IC-RAS-02)
- ret_addr comes from ras_fall_through_p2[s] (FTB fallthrough
  address). RAS does not compute PC+2/+4.
  (ras_decisions.md section 8)
- Same-cycle bypass: slot0=call and slot1=return in the same
  cycle, slot1 pop target forwarded from slot0 push data
  combinationally. (ras_interfaces.md IC-RAS-04)
- ras_restore_val takes priority over p2 push/pop same cycle.
  (ras_interfaces.md IC-RAS-09)
- One commit event per cycle. FTQ retires one entry per cycle.
  (ras_interfaces.md IC-RAS-10)
- p3/s3 repair: inverse operation when p3 FTB prediction
  disagrees with p2 operation applied. push->pop and pop->push
  within one p2/p3 pair cannot occur.
  (ras_decisions.md section 1, ras_interfaces.md IC-RAS-11)

## Assumptions made not explicit in the prompt
- Slot count: ports are sized [0:NUM_PRED_SLOTS-1] (=2). The
  dual-slot sequential model (slot 0 before slot 1, with bypass)
  is written as a slot loop and is correct for the documented
  2-slot configuration.
- Signal naming: the prompt requirement steps use p-stage names with
  slot array indexing (e.g. ras_fall_through_p2[s],
  ras_tos_valid_p0[s], ras_br_type_p3[s]). The interface doc
  section 3 uses s-stage scalar names (s2_*_p0/p1). Because the
  Background note mandates p-stage in all RTL, and keeping _p0/_p1
  slot suffixes would collide with p0/p1 stage names, ports use the
  requirement-step naming: ras_<base>_p<stage> as unpacked arrays
  indexed by slot [s]. This satisfies the section 4 contracts
  (IC-RAS-01..11) behaviorally.
- s3 repair ports: IC-RAS-11 lists s3 repair inputs that section 3
  notes are not yet in the port list (RI-4). Step 12 requires p3
  repair, so ras_pred_val_p3[s] and ras_br_type_p3[s] are included.
- Commit-stack pointer convention: CSP is a free (write) pointer;
  top entry is at CSP-1; empty when CSP==0. The snapshot carries
  only spec pointers, so CSP convention is internal and free to
  choose. Commit overflow uses natural mod-32 wrap (graceful
  degradation, no fault) per decisions 3.3.

## Decisions made not explicit in the prompt
- BOS on commit: set to ras_commit_snapshot.tosr on a call or
  return commit. Rationale: in-order commit makes the committed
  entry's recorded post-op top the new committed boundary. Step 13
  says only "Advance BOS from ras_commit_snapshot"; the exact field
  is unspecified. Flagged for validation at bp_cluster integration.
- p2/p3 coexistence: in a pipelined RAS, p2(N) and p3(N-1) update
  the same spec pointers in one cycle. The scan computes a single
  combinational next state, applying p3 repair first (correcting the
  prior op) then this cycle's p2 ops. Snapshots capture post-p2
  working pointers. Exact pipeline interleaving to be validated at
  integration (ties to RAS-3).
- p3 repair push address: re-pushes the registered p2 fallthrough
  (p3_fallthr_q[s]); a missed-push or undo-pop re-push has no other
  address source at this unit boundary. Approximation documented for
  integration review.
- Commit-stack recursion counter is written to 0 on push (not
  incremented on matching commit). decisions section 5 mentions
  commit rctr update on recursive commit; step 13 does not require
  it. Deferred; commit_rctr field is present and reset-cleared.
- restore vs commit on BOS: restore wins (mispredict recovery
  overrides). restore is pointer-only and does not clear data
  (decisions 4.3).
