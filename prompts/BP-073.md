## Claude.ai Assessment

Good result, and the IA was honest about the one thing that didn't fully land — which is exactly the part you need to read carefully, because "ALL EXTERNAL ANCHORS MATCH" overstates what was actually proven. Drop this in the Claude.ai Assessment block if it's useful.
Bottom line: BP-073 closed the implementation-fidelity gap, not the geometry-correctness gap — and it couldn't close the latter, because the docs don't contain the independent definition that would. The hash-rule docs specify only how a fold is consumed (tmpA = (PC >> off) ^ fh), never how fh is computed from the GHR. So there was no table-side fold definition to anchor against, and the IA correctly hit the fallback I wrote into Binding Decision 3: compute from the Xiangshan definition and flag the doc gap. That's the right behavior, but it has a consequence the IA states plainly in its Decisions section — the fallback reference necessarily shares the DUT's posmap, so the DUT==reference comparison does not catch a wrong posmap. The anti-drift guarantee rests entirely on the three frozen literals, not on a geometrically independent computation.
So here's what actually got proven, precisely:
The xs_fold_ref is a genuinely independent implementation — age-indexed, never reads ghr_buf, never reconstructs the circular index, doesn't call the DUT fold functions, isn't the BP-072 fold_ref. That catches the class of bug BP-072 surfaced (addressing/pointer-walk/off-by-one), and catching it is real value. The three literals are hand-derived fixed constants that freeze the geometry against future drift. I checked all three derivations by hand and they're correct — TC14 (0xE5: ages {0,1,3,6,7} → bits {7,0,2,5,6}), TC15 (0x9), and TC16 (0xC000_0000_0000_0000: the GHR wrap and 64-bit boundary are both genuinely exercised, bits 62/63 unrepresentable in 32b). The arithmetic is sound and the test construction is clean.
What it does not establish: that the posmap itself is the bit order the tables actually need. The literals were computed from the same posmap the DUT uses, so if posmap were wrong, the IA would have hand-derived the same wrong value the DUT emits and the check would still go green. The literal freezes the geometry; it doesn't independently validate that the geometry is correct. The real external anchor for correctness isn't in BP-073 at all — it's the session-054 consultation of the Xiangshan source (for the tables) plus the 071/072 consultation of the same source (for bp_history). Both are real, but they were checked against the source at different times and never against each other end-to-end — bp_history's actual output bits have never flowed into an actual table hash to produce a known index in simulation.
That residual gap is a producer→consumer end-to-end test, and it legitimately belongs at bp_cluster integration, not at the unit bench — so this isn't alarming, it's the normal boundary of unit verification. But I'd record it explicitly rather than let it ride on the "all anchors match" headline. Two things to track:

It pairs naturally with TD #69/#70 (rollback recompute at cluster) but is a distinct property — those are about recovery, this is about producer/consumer format agreement.

## Hypothesis

fold value computed independently from the table hash-rule docs'

## Background
BP-072 (session-055) landed the merged module-owned-pointer +
Xiangshan-fold bp_history.sv and proved, in simulation, that the
DUT's incremental fold equals its rollback recompute against ONE
testbench-local reference fold_ref. That proof is necessary but not
sufficient: fold_ref was written to the same geometry the DUT uses
(posmap(i) = (i+W-1)%W, newest at the high end). A geometry error
shared by the DUT and fold_ref would satisfy DUT == fold_ref and
report green. The dual-slot equivalence (TD #74) is therefore proven
self-consistently, but nothing yet proves the fold VALUE bp_history
emits is the value the TAGE/ITTAGE/SC tables actually consume.

session-054 established (and checked against the Xiangshan source)
that the TAGE/ITTAGE table hashes are Xiangshan-faithful, and that
only bp_history had the wrong fold bit order. The table hash-rule
docs are the table-side authority for how a fold is defined and
consumed. They are the correct EXTERNAL reference for this check:
a fold value built from the hash-rule definition is independent of
bp_history's fold_step / fold_ghr and of the BP-072 fold_ref.

This task is verification-only. bp_history.sv is read-only (do not
modify the RTL geometry; this task tests it, it does not change it).
The reference here is independent in a different sense than BP-072's:
BP-072 required independence from DUT INTERNALS (read ghr_buf, not
fold_step); this task additionally requires independence from the
DUT GEOMETRY (derive the expected value from the table-side hash-rule
definition, not from the same posmap the DUT uses).
