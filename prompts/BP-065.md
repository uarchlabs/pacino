## Hypothesis
A single discrete 1R1W FTB storage array with tree-PLRU bit storage
and read-old-on-collision can be implemented lint-clean, sized at
FTB_ENTRY_WIDTH=108 / FTB_SET_WIDTH=432, and remains substitutable for
a 1R1W SRAM or register file without any change to its port list.

## Background
FTB is the s2 second-level BTB (one array, one lookup per cycle, one
entry per 32-byte block). See ftb_decisions.md sections 2.1, 2.4, 3,
8, 8.1 and ftb_interfaces.md sections 3 and 5. This task creates only
the storage array and the package parameters it needs. Field semantics
(valid/tag/br0/br1/jump/fallthrough, way match, PLRU victim selection
and update) are NOT in the array; they are in ftb_cntrl (BP-066). The
array stores FTB_SET_WIDTH bits per set, returns the whole set on read,
and writes one selected way per write.

## Binding Previous Decisions
- Package parameter naming. Define the FTB widths as FTB_ENTRY_WIDTH
  and FTB_SET_WIDTH (FTB_-prefixed) to avoid colliding with any
  existing global symbol. The bare "ENTRY_WIDTH" / "SET_WIDTH" used in
  the planning docs map to these two names. All other names are exactly
  as the docs give them.
- FTB_WAYS is currently 8 in the package. This is a FIX to 4, not an
  add (ftb_decisions.md 8, ftb_interfaces.md 5). Nothing consumes FTB
  params yet, so this fix changes no existing module behavior.
- The array is field-agnostic. It does not decode valid/tag/br/jump,
  does not compute a way match, and does not compute the PLRU victim
  or the PLRU update. It only stores and returns raw bits. wr_way
  arrives already one-hot from ftb_cntrl; the array applies it.
- Read-old-on-collision: a read and a write to the same set in the
  same cycle return the OLD (pre-write) set contents on rd_data
  (ftb_interfaces.md 3, F19).
- PLRU storage only: the array stores PLRU_BITS per set with a
  read/modify/write port set (plru_rd_*, plru_wr_*). It returns the
  stored bits and writes plru_wr_data when plru_wr_en is asserted.
  The victim computation and the next-PLRU-state computation are in
  ftb_cntrl, not here (ftb_decisions.md 5.3).

## Assumptions made not explicit in the prompt
- rd_en gating: rd_data is the addressed set when rd_en is asserted,
  else all-zero. The port list names rd_en ("1 = read this cycle") but
  does not state the not-read output; zero is the cleanest defined,
  substitutable behavior and ftb_cntrl qualifies reads with the FTB
  valid/hit decode anyway.
- Synchronous reset clears both memories to 0. The prompt says
  "synchronous write" and "synchronous reset"; clearing the array on
  rstn keeps a cold FTB from presenting residual contents. The array
  stays field-agnostic -- it does not interpret the zeroed bits.
- plru_rd_data has no read enable in the port list, so it is presented
  unconditionally (combinational read of the PLRU store).
- Way packing: way 0 occupies the low FTB_ENTRY_WIDTH-bit slice of the
  set; way w is bits [w*FTB_ENTRY_WIDTH +: FTB_ENTRY_WIDTH]. The array
  is opaque, so the packing order is internal and consistent between
  the (whole-set) read and the (per-way) write.

## Decisions made not explicit in the prompt
- Read timing (prompt step 6 left it to the implementer): combinational
  read of the storage, NOT a registered SRAM read. Rationale: with a
  combinational read and a synchronous (NBA) write, a same-set
  same-cycle read returns the pre-write contents for free, which makes
  the read-old-on-collision rule (F19) explicit and removes a state
  element. The external p0->p2 contract is met by the FTB top
  (BP-067), so internal read latency is unconstrained here. A
  registered 1R1W SRAM remains a valid drop-in substitute.
- Parameter kind: base knobs (FTB_ENTRIES, FTB_WAYS, FTB_BLOCK_BYTES,
  TAR_STAT_BITS, FTB_BR_TGT_BITS, FTB_JMP_TGT_BITS, FTB_CONF_WIDTH,
  FTB_CONF_SUPPRESS_THRESH) are `parameter int`; derived widths are
  `localparam`. FTB_CONF_INIT is `localparam logic [FTB_CONF_WIDTH-1:0]`
  so the 3'b011 literal carries its declared width.
- Naming: the planning docs' bare ENTRY_WIDTH/SET_WIDTH are defined as
  FTB_ENTRY_WIDTH/FTB_SET_WIDTH per the Binding Decisions. PLRU bits
  use the spec name PLRU_BITS (the broken prior code's FTB_PLRU_BITS
  was removed, not kept as an alias). FTB_BLOCK_BITS and the duplicate
  FETCH_BLOCK_BYTES from the broken block were removed; FTB_TAG_BITS is
  now derived from FTB_OFFSET_BITS ($clog2(FTB_BLOCK_BYTES)), never from
  FETCH_BLOCK_BYTES.
