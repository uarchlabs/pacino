## Claude.ai Assessment

 \#62 closed, no RTL change, count ledger still consistent (87 + 8 = 95 on both init paths). The fold worked — TC-95 proves RAM-level write isolation at sim_tage (T1 allocated to 0x0509, T2 held at its 0x003F sentinel), so TAGE allocation has the selected-changed / non-selected-unchanged proof that ITTAGE had to wait until BP-055 Phase 4 for. The field-order cross-check came back clean: RTL writes {TAG, EPC, USE, CTR, VAL} matching the entry-format doc, so the BP-053 swap has no TAGE analog. TC-93's two-epoch contents check (0x0309 vs 0x0389) confirms EPC=epoch in the allocated entry, which also lightly re-touches the aging-EPC interaction from a different angle.

One finding in the results is worth pulling up to the front, because it changes a label rather than just confirming one. The author reports that the "no-consecutive skip" I asked for in step 3 does not exist as such — the RTL guard is alc_comp_p1[s] == 0, which means the scan stops at the first candidate, not that a selected table causes the next to be skipped. TC-90 verifies the actual behavior (T2 allocated, T3 untouched) and the author notes it's "functionally equivalent" for that stimulus. That's the spot I flagged last turn as the thinnest mirror assumption, and it played out: I carried "no-consecutive skip" over from the ITTAGE BP-053 description, and TAGE's policy is first-match-wins. Two consequences:

The test is valid — first-qualifying-table selection is proven, which is the real policy. But "functionally equivalent" is true only for the seeded case (one candidate). A literal skip-next policy and a stop-at-first policy diverge when there are two allocatable candidates in a row: skip-next would pass over the first and take the second; stop-at-first takes the first. TC-90 seeded T2 as candidate and T3 as a distinct non-candidate residue, so it can't distinguish the two policies — it only shows T3 wasn't written, which both policies predict. So the allocation selection among multiple consecutive candidates isn't actually pinned by this run. Low risk, because the RTL is unambiguous (the author read it), but if you want the policy itself proven rather than inferred from the source, that needs a two-consecutive-candidate stimulus. Worth a note for the #71 capstone or as a small follow-on, not a re-run.

And the label should be corrected where it lands in STATUS: TAGE allocation is "scan stops at first allocatable table > provider," not "no-consecutive skip." If the Draft alloc-rules doc says "skip," that's a doc/RTL wording mismatch the author didn't flag as BUG-001 (probably because it's a description difference, not a behavioral one) — but when Jeff promotes the doc to Complete, that row should be reworded to match the stop-at-first RTL so the next reader isn't misled the way I was.

## Hypothesis

On a mispredict that meets the allocation trigger, TAGE selects
a table longer than the provider whose entry is allocatable
(u_eff==0), per tage_cntrl_alloc_rules.md, and writes that one
table's RAM with the allocation init contents (VAL=1, weak CTR,
USE=0, EPC=epoch, TAG=alc_tag) at the pre-hashed alc_idx. When
the scan finds no candidate, alc_comp==0 and the write is
suppressed. The selected table's RAM changes and a non-selected
table's RAM does not.

## Background

#62 is the TAGE half of the allocation proof. The ITTAGE half
(#63, BP-053) confirmed 13 allocation rules with no RTL change,
but ran at sim_ittage_cntrl with no table instantiated, so it
proved only that the controller emits the correct per-table
selector -- not that the write lands in the selected RAM and
nowhere else. BP-055 Phase 4 later closed that for ITTAGE at
sim_ittage. This task runs at sim_tage with the tables present,
so it proves the RAM-level write isolation in the same session.

#66 (TAGE structural rework) is the prerequisite for #62 and is
closed (BP-045): the alloc and update index buses were collapsed
to shared per-slot buses. Use the post-rework signal names
(t_alc_index_u0, t_alc_tbl_sel_u0, t_alc_wd_u0, and the related
shared buses) as they exist in tage_cntrl.sv.

tage_cntrl_alloc_rules.md is Draft. This task is the directed
proof of each row; promotion of the document to Complete is a
separate post-session action. If the Draft document and the RTL
disagree, stop and report the disagreement rather than silently

BP-053 found the ITTAGE alloc-rules write-data field order

## Specific Requirements

Read each row of tage_cntrl_alloc_rules.md before encoding its
test. Use tage_table_entry_formats.md to set seed fields and to
decode the allocated-entry readback. With aging disabled, an
entry's allocatability is controlled directly by its raw USE
(USE==0 -> u_eff==0 -> allocatable); use this to seed candidate
and non-candidate entries without invoking the aging path.

1. Trigger gating. Confirm allocation occurs on a mispredict
   that meets the trigger (provider not the longest table, a
   longer table allocatable), and does NOT occur when the
   trigger is absent (correct prediction, or provider already
   the longest table). On the no-trigger case, confirm no table
   RAM changes.

2. Which table allocates. Seed allocatable (USE==0) and non-
   allocatable (USE>0) entries across T1-T4 and confirm the
   table the rules select is the one written. Confirm T0 is
   never selected.

3. No-consecutive skip. Exercise the allocation-throttling skip
   policy per the rules (a selected table causes the next to be
   skipped) and confirm the selection matches.

4. No-candidate sentinel. Drive a mispredict where no longer
   table is allocatable. Confirm alc_comp==0 and that no table
   RAM is written.

5. Pre-hashed index. Confirm the allocation writes at the
   pre-hashed alc_idx, derived from the prediction-time hash via
   the existing seed-and-predict pattern in tb_tage.sv.

6. Allocated contents. Decode the allocated entry and confirm
   VAL=1, the weak/reset CTR value, USE=0, EPC=epoch, and
   TAG=alc_tag, per the alloc-rules init and the entry format.
   Confirm the write-data field order matches
   tage_table_entry_formats.md.

7. alc_we gating. Confirm the allocation write-enable asserts
   only under the trigger-and-candidate condition and is
   deasserted otherwise.

8. RAM-level write isolation (folded from the #71 capstone).
   On a triggering allocation, read back: (a) the selected
   table entry changed to the expected allocated contents, and
   (b) a non-selected table entry, seeded to a known non-reset
   value, is unchanged. Check (b) must compare against the
   seeded value so a spurious write to a non-selected table is
   detected.
