The t0_dec_min_sat_tst catch is the best find in the set — that wasn't a wrong expected value, it was architecturally impossible stimulus (diff=1 with both providers on T0, violating Constraint 2), the kind of thing that passes by accident and tests a state that can't occur.

The one item that deserves your eyes is the architectural flag Claude Code raised in Other Notes. It observed that the HAND-FIX-003 T0 semantics — INC when correct, DEC when wrong — make the T0 counter a confidence counter, not a direction counter. That's accurate: row 13a (not-taken, correct) increments, where a standard bimodal would decrement on a not-taken outcome. If the T0 entry is genuinely "2b CTR only" with no separate direction bit, then the direction source has to live elsewhere — presumably tage_bim.sv, which the status tracks as a distinct module. This is very likely intentional (you authored HAND-FIX-003 deliberately, BUG-001 is recorded, and it was manually validated), so I'd treat it as something to confirm and record in the Claude.ai Assessment section rather than reopen. But it's worth confirming explicitly, because every test now built on that rule inherits the assumption.

Two smaller flags. The missing-coverage dispositions rest on a "same code path → no new test" assertion that wasn't verified against the RTL structure — defensible and within the prompt's "don't duplicate VALID coverage" instruction, but it's exactly the assumption that hid the stale expected values in the first place. And the header Status checkbox is left unticked despite the console reporting complete; minor, but this project's tooling parses those boxes.
