## Hypothesis

With tage_enable_aging high, the epoch counter (lcl_epoch)
advances when the interval counter reaches its boundary, wraps
within its width, and the per-entry age = (lcl_epoch - EPC)
within the epoch width drives a reduction of USE to u_eff. An
aged entry's u_eff falls below its raw USE and reaches zero at
sufficient age, making it allocatable where a fresh entry is
not. With tage_enable_aging low, the epoch is frozen and no
aging occurs.

## Background

#60 is the TAGE half of the aging proof. The ITTAGE half (#61,
BP-052) confirmed seven aging rules with no RTL change, forcing
the interval and epoch counters hierarchically rather than
walking the full interval, and observed that the epoch advances
the tick after the interval reaches its boundary (an N+1
periodicity, negligible). This task applies the same method to
TAGE.

Every prior tage test ran tage_enable_aging=0, so this path has
never been exercised. The u_eff reduction was observed
incidentally in BP-055 and BP-056 (an aged entry's u_eff fell
below raw USE), but never directed-tested as the unit under
test.

The u_eff/USE-decrement rules are in the aging section of
tage_cntrl_use_update_rules.md. The epoch-advance, epoch-wrap,
and enable-gating mechanism may not have a documented rule
beyond the RTL. If any aging behavior you test lacks a
documented authority to cite, report the gap rather than
inventing a citation.

## Assumptions made not explicit in the prompt

1. The forced boundary (lcl_aging_interval=0) matches the
   natural counter state when tage_aging_interval=0: on reset
   the counter is loaded with tage_aging_interval. With
   tage_aging_interval=0 the counter is already at 0 on the
   first pred_rdy_p2. Forcing achieves the identical state
   without natural walking, consistent with BP-052 approach.

2. The TAGE N+1 epoch-advance timing matches ITTAGE (BP-052).
   The aging_ff and pred_val_p2 pipeline register are both
   clocked at posedge clk in separate always_ff blocks.
   At posedge C (pred_val_p2 registers to 1 via NBA) the
   aging_ff samples the OLD pred_val_p2=0. At posedge D
   the aging_ff sees pred_val_p2=1 and advances the epoch.
   TC-83 explicitly checks epoch=0 at posedge C and epoch=1
   at posedge D to verify this.

3. The u_eff alloc scan reads EPC/USE from the RAM regardless
   of tag match. VAL=0 entries still contribute their EPC and
   USE to the u_eff computation for the alloc scan. This was
   confirmed from tage_cntrl.sv gen_ueff generate block which
   reads t_cntrl_bits_p1 (raw RAM output, not gated by hit).

4. After TC-82 re-asserts rstn, a while(!tage_rdy) poll is
   used so that sram_init completes before any backdoor-written
   entries are used. On the fast path tage_rdy=1 immediately;
   on the slow path the poll drains ~2048 init cycles.

## Decisions made not explicit in the prompt

1. TC-82 re-asserts rstn to test the register loading directly.
   The alternative (checking post-initial-reset values at the
   start of test_body) would require restructuring earlier tasks.
   Re-reset is cleaner and exercises the same logic path.

2. Entry encoding verified analytically before any simulation:
   - 0x003B = TAG=00 EPC=00 USE=11 CTR=101 VAL=1 (TC-85 T1)
   - 0x000F = TAG=00 EPC=00 USE=00 CTR=111 VAL=1 (TC-86 T1)
   - 0x0030 = TAG=00 EPC=00 USE=11 CTR=000 VAL=0 (TC-86 T2-T4)
   - 0x002B = TAG=00 EPC=00 USE=10 CTR=101 VAL=1 (TC-87 T1)
   Field layout (T1-T4): bit0=VAL bits[3:1]=CTR bits[5:4]=USE
   bits[7:6]=EPC bits[15:8]=TAG. Confirmed from tage_table.sv
   VAL_LSB=0 CTR_LSB=1 USE_LSB=4 EPC_LSB=6 TAG_LSB=8.

3. TC-87 sub-test A uses USE=10 (not 11) to distinguish aging-
   disabled u_eff=USE from a possible age-1 result of 01. This
   ensures the check is discriminating.

4. tage_aging_interval set to 32'h0 as the default throughout
   (enables natural boundary matching for reachability). Forcing
   used only to set counters to known values at test start.

## Documentation Gaps

1. Epoch wrap: the 2-bit modular wrap of lcl_epoch (11->00) is
   not stated in tage_cntrl_use_update_rules.md. It is implicit

   from having two separate always_ff blocks clocked at the same
