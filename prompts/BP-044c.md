The hypothesis predicted the bug and the fix is the right shape.

The bug is exactly where the prompt aimed: use_we (and epc_we) gated on prm_match only, so for the UP=0 rows the alternate table — where prm_match=0 — silently dropped the USE write, while the primary table would have written USE at the alt's index, the wrong address. Two failure modes from one gate. The fix gates on (prm_ctr_wr & prm_match | alt_ctr_wr & alt_match), reusing the already-correct CTR strobes as the provider selector. That's minimal and structurally right — it makes the USE write follow the same provider split the CTR path uses, which is the correct invariant. The Other Notes diagnosis ("USE reused the simpler prm_match gate from the CTR-target path") is a credible root cause, not just a description.

This also validates the TD #51 thesis directly. BP-039 found the CTR inversion; TD #51 warned the USE/TGT/allocation blocks likely carried analogous errors. CTR (044b) and now USE (044c) both confirmed. That's two of the four blocks TD #51 named. TGT and allocation are still unaudited and are now the standing suspects — same backdoor method applies, and given two-for-two, I'd treat them as probable rather than possible.

Coverage is honest this time: all 6 rows, UP=0 proven with a real second-table alternate (IT2 primary, IT1 alternate, observed alt_useful 1→2 and 2→1), saturation both providers both ends, readback through prediction. 81 checks.

Two things to weigh, neither blocking:

The epc_we fix is correct but slightly outside the letter of the prompt. EPC rides alongside USE (same gate, written together), so fixing use_we without epc_we would have left EPC writing to the wrong table on UP=0. The IA was right to fix both. But EPC has no test coverage here — the fix is reasoned, not demonstrated by readback. Worth noting that EPC's UP=0 write path is now changed but unproven. The aging path that consumes EPC is also unexercised (tests run aging=0), so this won't bite until aging integration — but it's an untested edge in changed RTL.

The alloc-residue invalidation is back (IT3 at PC=0x300, IT2 at PC=0x500, hand-computed indices), which you've already accepted. Same load-bearing-but-fragile infrastructure as 044b; the index assumptions are documented in Assumptions.

State of ITTAGE update paths now: CTR fixed and fully proven (044b), USE fixed and fully proven (044c), EPC fixed as a USE rider but unproven, TGT and allocation unaudited and — on the TD #51 pattern — likely carrying the same provider-gating defect. That's the natural next thread if you keep going: audit TGT and allocation write paths the same way, since the method and the testbench infrastructure now exist to do it cheaply.

defect analogous to the CTR strobe swap fixed in BP-044b.
The USE write-enable in ittage_table.sv (use_we) gates on
prm_match only, not alt_match. If the USE rule for the UP=0
alt-provider rows requires writing the alternate entry's USE
field, that write may never land. The UP=0 USE rows have not
been exercised before.

tb_ittage.sv in BP-044b. It writes a full entry

prediction in the cntrl_bits USE field. A genuine alternate
provider requires two tables seeded at the same PC: longer
table primary, shorter table alternate.

### ittage_table.sv (we_s0, we_s1 blocks) -- DEFECT

Pre-fix:
  use_we_s0 = ittage_upd_val_u0[0] & use_wr_u0[0]
            & prm_match_s0;

  prm_match_s0 = (prm_comp == THIS_TBL_SEL). For UP=0 rows
  (5-6), the alt provider's table has prm_comp pointing to
  the longer table. The alt table's prm_match=0, so
  use_we never asserts and the write is silently dropped.
  Simultaneously the primary table (where prm_match=1) would
  write USE at alt_idx, the wrong address.

Post-fix (Table 7 rows 3-4 UP=1, rows 5-6 UP=0 cited):
  use_we_s0 = ittage_upd_val_u0[0] & use_wr_u0[0]
            & (prm_ctr_wr_u0[0] & prm_match_s0
             | alt_ctr_wr_u0[0] & alt_match_s0);

  alt_match_s0 = (alt_comp == THIS_TBL_SEL). When UP=0,
  alt_ctr_wr=1 and alt_match=1 in the alt table -> write
  lands. prm_ctr_wr=0 in all UP=0 cases -> primary table
  does not write USE at the wrong address.
  Same fix applied to epc_we_s0, use_we_s1, epc_we_s1.
