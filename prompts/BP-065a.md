# Task Overview

Finalization of the FTB storage split (follows BP-065). Move the
entry-valid bits and the tree-PLRU state out of ftb_array into a new
ftb_plru flop module, leaving ftb_array as pure 1R1W data RAM. Flip all
storage enables to active low to match the BPU array convention. This
supports eventual SRAM migration: the data array becomes a substitutable
SRAM macro with no reset, and validity is held in resettable flops
(ftb_plru) -- the FTB has no sram_init mechanism. PLRU victim and
next-state COMPUTE stays in ftb_cntrl (BP-066); ftb_plru is storage
only. Lint only this task; the FTB sim suite is built against the top
in BP-068.

## Hypothesis
The FTB storage can be split into pure 1R1W data RAM (ftb_array,
FTB_RAM_ENTRY_WIDTH=107 / FTB_RAM_SET_WIDTH=428, no reset) plus a
resettable flop module (ftb_plru) holding the entry-valid bits and the
tree-PLRU state, both with active-low enables, lint-clean, with no
regression to the shared-package suites.

## Background
BP-065 built ftb_array with the entry-valid bit inside the entry, the
PLRU bits inside the array, active-high enables, and a reset that
cleared the array. This task finalizes the structure for SRAM
migration per the updated ftb_interfaces.md (section 1 hierarchy,
section 3 ftb_array revised, section 3a ftb_plru new, section 5 params,
IC-FTB-12/13/14). The logical 108-bit entry is unchanged; it is now
partitioned -- 1 valid bit/way in ftb_plru, 107 in ftb_array. PLRU
victim selection, next-state, way-match, and valid set/clear logic stay
in ftb_cntrl (BP-066, not built yet); ftb_plru and ftb_array are
storage only.

tage_table.sv is loaded read-only as the reference for the BPU
active-low enable convention -- match its enable polarity and naming
token exactly.

## Binding Previous Decisions
- Storage split (IC-FTB-12). ftb_array becomes pure 1R1W data RAM: no
  entry-valid bit, no PLRU state, no compute. ftb_plru (new) holds, per
  set, the FTB_WAYS entry-valid bits and the PLRU_BITS tree-PLRU state,
  in flops. Neither storage module computes the victim, the next PLRU
  state, the way-match, or which way to allocate -- all of that is
  ftb_cntrl (BP-066). ftb_plru is flop storage with read and write
  ports only.
- Width partition. The logical entry stays 108 (FTB_ENTRY_WIDTH) /
  FTB_SET_WIDTH 432, documented as logical. ftb_array uses
  FTB_RAM_ENTRY_WIDTH = FTB_ENTRY_WIDTH - 1 = 107 and FTB_RAM_SET_WIDTH
  = FTB_WAYS * FTB_RAM_ENTRY_WIDTH = 428. The br0/br1/jump FIELD-valid
  bits stay in the RAM entry; only the ENTRY-level valid (1 bit/way)
  moves to ftb_plru. FTB-1 / IC-FTB-08 stay closed -- logical width
  unchanged, only partitioned.
- Active-low controls (IC-FTB-13). All enables on both storage modules
  are active low: ftb_array rd_en_n / wr_en_n, ftb_plru rd_en_n /
  val_we_n / plru_we_n. Match the exact polarity and naming token used
  by tage_table.sv; if that array uses a token other than "_n",
  reconcile to it and note the choice in Results Capture.
- ftb_array has NO reset (pure SRAM-style RAM). Remove rstn and the
  array reset-clear from BP-065. Cold validity is owned by ftb_plru.
  Do not rely on array power-up contents.
- ftb_plru reset (rstn, active low) clears all entry-valid bits to 0
  and resets all PLRU state to 0. This is the FTB cold init; there is
  no sram_init for the FTB.
- Read-old-on-collision on BOTH modules (IC-FTB-14). A same-cycle read
  vs same-set write returns the old contents, so the prediction sees a
  coherent pre-update snapshot of data + validity/PLRU together.
- Way-slice packing. ftb_array packs way w at
  [w*FTB_RAM_ENTRY_WIDTH +: FTB_RAM_ENTRY_WIDTH]; wr_way[w] selects
  way w. ftb_plru val_way is the same one-hot convention (bit w = way
  w).
- Separate valid and PLRU write ports on ftb_plru so an allocate can
  set valid and update PLRU for the same set in one cycle.

## Decisions made not explicit in the prompt
- Enable-token reconciliation: tage_table.sv uses the "_n" suffix token
  for active-low controls (wen_n on bw_ram, internal ram_wen_n_*) and
  rstn for active-low reset. The prompt's requested names (rd_en_n,
  wr_en_n, val_we_n, plru_we_n) already match the "_n" token exactly, so
  no rename was needed. Polarity: 0 = active (read/write this cycle),
  matching tage_table.sv wen_n.
- Resolved widths: FTB_RAM_ENTRY_WIDTH = FTB_ENTRY_WIDTH - 1 = 107;
  FTB_RAM_SET_WIDTH = FTB_WAYS * FTB_RAM_ENTRY_WIDTH = 4 * 107 = 428.
  FTB_ENTRY_WIDTH (108) and FTB_SET_WIDTH (432) retained as the LOGICAL
  widths with an added comment.
- ftb_plru read returns '0 when rd_en_n is high (deasserted), mirroring
  the ftb_array rd_en_n behavior for a consistent storage contract.
- rd_valid and rd_plru are driven by two separate continuous assigns
  (no internal dependency chain), which is acceptable per the project
  combinational-style rule.
