### 4.6 Basic direction: present implies taken

The conditional fields store no explicit direction bit (4) — only always_taken and conf. FTB's basic direction for a valid conditional is TAKEN: ftb_brI_taken_p2 = ftb_brI_valid_p2 (classic BTB presence semantics). always_taken (4.3) and conf (7) modify trust, not the direction. Because FTB tracks not-taken branches too (5.2), a reliably-not-taken branch keeps always_taken=0 and trains conf DOWN, so its conf stays below threshold, suppression never fires, and TAGE supplies the direction. ## Hypothesis
All FTB control behavior -- read/way-select, branch-type classify,
fallthrough reduce/reconstruct, carried-writeWay allocate/evict with
tree-PLRU, update field writes, and confidence train/suppress -- can be
implemented in one ftb_cntrl module that drives the ftb_array and
ftb_plru storage ports and the FTB top functional ports, lint-clean,
with no array re-lookup on the update path.

## Background
The FTB storage is split (BP-065a). ftb_array is pure 1R1W data RAM
(FTB_RAM_ENTRY_WIDTH=107 / FTB_RAM_SET_WIDTH=428, no entry-valid, no
reset). ftb_plru holds, per set, the FTB_WAYS entry-valid bits and the
PLRU_BITS tree-PLRU state in resettable flops. Both are storage only.
ftb_cntrl owns every behavior and drives both modules.

Authority: ftb_decisions.md (sections 1, 2.4, 3, 4, 4.2, 4.5, 5,
5.1-5.5, 8, 8.1), ftb_interfaces.md (section 2 functional ports cntrl
drives, section 3 ftb_array ports, section 3a ftb_plru ports, section 4
invariants incl. IC-FTB-10/12/13/14), and
ftb_confidence_override_rules.md (the conf counter: training, acting,
chicken bit). The FTB parameters and FTB_RAM_* widths are in
bp_defines_pkg.sv.

The cluster predicts two branches per cycle as br0 and br1 -- the two
conditional fields of the ONE indexed entry. There are no per-slot RAMs
and no slot-1 lookup (ftb_decisions.md 2.1, IC-FTB-04). Do not import
TAGE/ITTAGE per-slot or NUM_PRED_SLOTS conventions.

- always_taken set/clear (5.5). Set at allocate to the resolved

### 107-bit RAM entry layout (ftb_entry_t, private to ftb_cntrl)
Packed struct, declaration order = MSB..LSB. Section 8 order minus the
relocated entry-valid:

  field                                   bits   running
  tag                                      26      26
  br0 {valid,pos,tgt,stat,always_taken,conf}
       1 + 3 + 13 + 2 + 1 + 3  = 23        23      49
  br1 {same as br0}                        23      72
  jmp {valid,pos,tgt,stat,isCall,isRet,isJalr}
       1 + 3 + 21 + 2 + 1 + 1 + 1 = 30     30     102
  pftAddr (PFTADDR_BITS)                    4     106
  carry                                     1     107

Total = 107 = FTB_RAM_ENTRY_WIDTH. The within-way bit order is private
to ftb_cntrl; the only external contract with ftb_array is the way
slice [w*FTB_RAM_ENTRY_WIDTH +: FTB_RAM_ENTRY_WIDTH]. ftb_cntrl packs
upd_new into array_wr_data and unpacks each way slice of array_rd_data
through the same typedef, so the layout is self-consistent. No FTB
struct was added to bp_structs_pkg.sv (per constraint).

### Tree-PLRU bit assignment (PLRU_BITS = 3, 4-way)
Each node points TOWARD the victim subtree/leaf:

  st[0] = root  : 0 -> victim in {way0, way1}, 1 -> {way2, way3}
  st[1] = left  : 0 -> way0, 1 -> way1
  st[2] = right : 0 -> way2, 1 -> way3

Victim (plru_victim): if st[0]==0 -> st[1]?way1:way0 else
st[2]?way3:way2. Reset state 000 -> first victim is way0.
Mark-used (plru_touch w): steer every node on w's path away from w.
For ways {0,1}: st[0]=1, st[1]=~w[0]. For ways {2,3}: st[0]=0,
st[2]=~w[0]. Implemented with the 2-bit way index (no 4-way
hand-unroll). Way-match is a FTB_WAYS for-loop.

### array_*/plru_* port naming and chicken_bit_enable
Storage-facing ports carry distinct prefixes so the top can wire two
peers that both expose rd_en_n/rd_addr: array_* -> ftb_array, plru_* ->
ftb_plru (IC-FTB-12/13). chicken_bit_enable is a 1-bit module input
(source TBD at bp_cluster, confidence section 10), used only in the
ftb_suppress_dir_p2 acting term.

## Assumptions made not explicit in the prompt
- Shared read-port arbitration. ftb_array and ftb_plru each expose ONE
  read port. The update path needs the carried way's current entry to
  read-modify-write conf / always_taken / target / pftAddr on a hit,
  and the carried set's current PLRU state to compute the mark-used
  next state. So an active update (ftb_upd_valid_u0) BORROWS both read
  ports, addressed by the carried set index (NOT a tag re-lookup --
  IC-FTB-10 is honored: hit/way are carried, never recomputed). Update
  borrows have priority; a prediction in the same cycle self-bubbles
  (pred_grant_p1 = valid_p1 & ~upd_active, so its p2 outputs read
  invalid). Justified because FTB is already a bubble-costing s2
  structure and update-channel scheduling is an FTQ concern (IC-FTB-09,
  deferred to bp_cluster).
- "no array re-lookup on the update path" is read as no second TAG
  lookup / way-match on update (hit + way are carried). A direct,
  carried-way addressed READ of the entry for the read-modify-write is
  still required and used, because the update port carries no conf, no
  stored always_taken, and no stored target -- conf train (inc/dec) and
  always_taken clear-on-not-taken are intrinsically RMW.
- FTB basic direction. The conditional fields store no explicit "taken"
  bit (entry format 4). FTB predicts a present (valid) conditional as
  TAKEN (BTB presence semantics); ftb_brI_taken_p2 = ftb_brI_valid_p2.
  always_taken is the stronger bypass hint; conf measures how reliably
  the taken prediction holds. Consistent with the confidence training
  comparison (ftb_upd_ftb_dir_u0 vs resolved).
- Pipeline depth. p0 request registered to p1; p1 reads + computes
  combinationally; results registered to p2. Two-cycle latency
  (s0 send, s1 registered, s2 valid). The p1 way-match block reads the
  registered valid_p1 so it is nba_sequent, not stl_sequent (CLAUDE.md
  Verilator note) for the eventual BP-068 sim.
- conditional/jump target base = block start = the block-aligned PC
  (low FTB_OFFSET_BITS zeroed), both on encode (update) and
  reconstruct (predict).
- Target status (TAR_STAT_BITS): 00 fit, 01 overflow (forward), 10
  underflow (backward). fit when the sign-extended displacement
  reconstructs the target exactly. Status is stored but NOT consulted
  on reconstruction (consistent with the no-error-check stance, 4.5);
  it is metadata for the cluster.

## Decisions made not explicit in the prompt
- pftAddr reduce/reconstruct arithmetic. Reduce: end_off = resolved end
  - block_base; pftAddr = end_off[FTB_OFFSET_BITS-1:INST_OFFSET]
  (= end_off[4:2], the in-block expanded-instruction index)
  zero-extended to PFTADDR_BITS; carry = |end_off[VA_WIDTH-1:
  FTB_OFFSET_BITS] (the end crosses the 32-byte block boundary).
  Reconstruct: block_base + (pftAddr << INST_OFFSET) + (carry ?
  FTB_BLOCK_BYTES : 0). Lossless for in-block ends; carry restores a
  next-block end. INST_OFFSET (=2) from the package is the /4 expanded
  granularity. The 4th pftAddr bit is reserved per PFTADDR_BITS sizing
  (the doc's "+1 extension"); only 3 significant index bits are used.
- pftAddr/carry rewritten on EVERY update (the reduction is idempotent
  when the boundary is unchanged), avoiding a boundary-moved detector
  the update port gives no clean signal for. Result is identical to
  "recompute only on boundary move."
- conditional target written on every conditional resolve
  ("rewrite-if-changed" and "always write the correct encoding" yield
  the same stored value); jump target written on every jump resolve
  (5.5/IC-FTB-01).
- in-block position field (pos) written as 0. The update port (2.5)
  carries no in-block position, so pos has no source; it is stored but
  not currently sourced and not output. Flagged as a gap below.
- PLRU touch funnel realized implicitly: because an active update owns
  the read ports, pred_grant_p1 is 0 whenever upd_active, so the
  prediction-hit touch and the update/allocate touch are mutually
  exclusive on the single PLRU write port -- "update/allocate wins"
  falls out without an explicit drop, matching 5.3.
