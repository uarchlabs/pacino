TD #57. The ITTAGE target (TGT) write path has no directed test.
Target replacement on misprediction has never been verified by
readback. The decisions document was just revised (section
"Concurrent CTR and TGT Writes") to relax the prior CTR/TGT

using backdoor RAM seeding and readback in tb_ittage, for both

the RTL if it does not conform.

## Claude.ai Assessment

You already caught it (Comment #2); I'm confirming it's correct and that the "harmless" framing is wrong.

The defect: when UP=0 and the target is written, tgt_we in ittage_table.sv gates on (prm_match | alt_match), so both the alternate (provider, IT1) and the primary (non-provider, IT2) get their target field overwritten with resolved_target. Per Seznec, only the provider's target is replaced. Writing the non-provider's target is wrong.

The IA's own words give it away. Assumption 4: "an existing behavior not prohibited by the spec." Other Notes: "IT2's incidental TGT update has no observable effect on correctness." Both are rationalizations, and both are false:

It is prohibited by the spec — your decisions document says the provider's target is replaced; it does not authorize writing the non-provider's. "Not explicitly forbidden" is not "permitted," and the IA inverted that.

It is not harmless. IT2 is the primary — the longer-history, higher-priority table. On the next prediction at that PC, IT2 hits and is selected as provider over IT1. Its target now holds a value written under the wrong history length. That is a corrupted higher-priority entry that will be consulted first. Calling it "incidental" with "no observable effect" is exactly the plausibility-over-proof failure this project keeps hitting — and the test didn't catch it because TC-TGT-D only reads back IT1, the value it expected to change, and never checks that IT2 was left alone.

So the test has the same blind spot as the original TC-UPD-06: it doesn't read the field that proves the no-write. A correct TC-TGT-D reads back IT2's target and asserts it is unchanged.

BP-049 is therefore not complete. TD #57 is not closed. The status should be in-progress, and BP-049a fixes it:

RTL: tgt_we must gate the target write on the provider match only — prm_match under UP=1, alt_match under UP=0 — not (prm_match | alt_match). This is the same class of provider-gating defect as the CTR and USE bugs in BP-044b/c; the target write path was simply never audited for it, which is precisely what TD #57 existed to do.

Test: extend the UP=0 cases to read back the non-provider (IT2) target and assert it is unchanged. Without that check the fix isn't proven — and the existing test passing proves nothing about the non-provider, by construction.

## Hypothesis

The ITTAGE target write path conforms to the revised
specification: on misprediction the target is replaced if and
only if the provider CTR is zero, and the CTR is decremented
(saturating at zero) when non-zero, with no target write. The
behavior holds for both the primary-provider (UP=1) and
alternate-provider (UP=0) cases. Directed tests with seeded RAM
entries and readback confirm the target is replaced exactly in
the CTR-zero case and left unchanged in the CTR-non-zero case.

## Background

TD #57: target replacement is untested. The target is written on
misprediction when the provider CTR is zero. The CTR/TGT
interaction is specified in ittage_cntrl_decisions.md, section
"Concurrent CTR and TGT Writes":
- Non-zero CTR on misprediction: decrement CTR, no target write.
- Zero CTR on misprediction: write the target field.
- The earlier mutual-exclusivity requirement is relaxed; a
  zero-to-zero CTR write is a no-op and is permitted.

The load-bearing condition is: the target is replaced when, and
only when, the provider CTR is zero on misprediction. The
relaxation is sound only if the target write is gated on the
CTR-zero condition directly, not inferred from the CTR write
being deasserted.

The backdoor RAM-seeding method established in BP-044b is the
test mechanism. Entry packing for the table RAM: VAL[0],
CTR[3:1], USE[5:4], EPC[7:6], TGT[45:8], TAG[ALLOC_DATA_WIDTH-1:
46]. Address split: bank = idx[INDEX_BITS-1], ent =
idx[INDEX_BITS-2:0]. The bw_write task in tb_ittage.sv writes a
full entry to u_ram_s0/u_ram_s1 inside the generated table.

## Binding Previous Decisions

- ittage_cntrl_decisions.md section "Concurrent CTR and TGT
  Writes" is the specification for target write behavior. Where
