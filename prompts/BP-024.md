## Step 1 -- alc_en gating conditions
The per-slot allocation enable is u_alc_wr[s] in
alc_upd_comb (tage_cntrl.sv lines 807-827).
Four conditions must ALL be true:
  (1) tage_upd_val_u0[s]  -- update valid, TB-driven
  (2) u_mispredict[s]     -- mispredict, TB-driven via meta
  (3) u_prm_comp[s] < TAGE_MAX_TBL(4)  -- TB-driven via meta
  (4) u_alc_comp[s] != '0  -- RTL (round-trip) or TB
                               (synthetic meta, must be set)
Additional undocumented gate on RAM write strobe
(tage_cntrl.sv lines 887-890):
  t_alc_wr_u0[t][s] = trx_type && u_alc_wr[s] && match
  trx_type = arb_grant_upd (tage.sv line 262), RTL-internal,
  generated by BP-023b credit arbiter, NOT TB-driven.

## Step 2 -- ueff scan logic
ueff_p1[t][s] computed in gen_ueff (lines 402-426):
  age = lcl_epoch[s] - t_epc_p1[t][s]  (2-bit modular)
  age==0 -> ueff = USE
  age==1 -> ueff = USE >> 1 (MSB only)
  age>=2 -> ueff = 0 (always a candidate)
TAGE_SRAM_INIT_VALUE=0 + FAST_INIT: all entries start
USE=0, EPC=0, epoch=0 -> age=0, ueff=0 for every entry.
Candidate scan (lines 532-541) walks t = prm_comp+1 ..
TAGE_NUM_TABLES-1 and takes the first t with ueff==0.
Root Cause B ruled out: no USE increment fires on any
allocation target entry before the first mispredict in
TC-42 or TC-43.

## Step 5 -- Root cause diagnosis
ROOT CAUSE: C
The specific gate that was always false:
  u_alc_comp[s] != '0  (tage_cntrl.sv line 816)
All prior synthetic-meta update tests (TC-01 to TC-41
except upd_alloc_tst) initialized meta with:
  meta = '0;
  meta.tage_prm_comp = <table>;
  meta.tage_mispredict = 1;
  // tage_alc_comp never assigned -- remains 3'b000
With tage_alc_comp=0, condition (4) is permanently false
regardless of what provider or USEFUL values are present.
Cause A ruled out: prm_comp was not always T4.
Cause B ruled out: USEFUL was not nonzero before first
mispredict (FAST_INIT + no USE write to target entries).
upd_alloc_tst (line 7402, line 1022) correctly sets
meta.tage_alc_comp = 3'd2 and should exercise the write
path -- it was added after the INFRA-005 baseline.

TC-42, TC-43, and upd_alloc_tst should produce hits on the
allocation write path. If zero counts persist, the issue
is trx_type not asserting during the update cycle; fix by
ensuring the update issues when no concurrent prediction
is pending (arbiter Rule 6 fires arb_grant_upd=1 for
standalone update, triggering trx_type=1).
