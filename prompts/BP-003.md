### Hypothesis

A 256-entry 4-way associative uBTB implemented in SystemVerilog,
with PC-indexed lookup and partial-tag matching, can produce a
valid s1 prediction (next-PC, taken bit, branch type) with
combinational output from registered SRAM contents, and can
accept a post-execute update in the same cycle as a new
prediction without structural hazard, when the update and
lookup address the same set.

---

### Background

The uBTB is the first predictor in the BP cluster pipeline. It
fires at s1, one cycle after the PC is presented at s0. Its
output is the initial next-PC fed to speculative fetch. If the
uBTB misses, no prediction is generated and fetch proceeds
sequentially (PC + fetch_width). The uBTB does not generate a
redirect signal -- it provides or withholds an initial
prediction only.

The Loop predictor is co-staged at s1 but is a separate module
(BP-004). This module is uBTB only.

At s2, FTB/TAGE may disagree with the uBTB result and fire
s2_redirect. The uBTB is the lowest-priority predictor in the
conditional branch override chain.

---

### Binding Previous Decisions

1. NUM_PRED_SLOTS is an elaboration-time parameter (1 or 2).
   The uBTB must support both configurations. Parameterize
   prediction input/output arrays on NUM_PRED_SLOTS.

2. Update policy: post-execute, not retire. One update channel
   per active prediction slot. When NUM_PRED_SLOTS=2, there
   are two independent update channels: upd[0] and upd[1].

3. uBTB miss behavior: pred_valid[slot]=0 for that slot.
   Fetch proceeds PC+fetch_width. No redirect signal generated
   by this module.

4. bp_pkg.sv defines all shared structs and parameters. Read
   bp_pkg.sv before writing any new localparams or structs.
   Do not redefine anything already present there.

  UBTB_BR_INDIR  = 3'b011  // indirect non-return (JALR)
  UBTB_BR_CALL   = 3'b100  // call: rd = x1 or x5
  UBTB_BR_RET    = 3'b101  // return: rs1 = x1 or x5
  // 3'b110, 3'b111 reserved

New structs:

  // One uBTB storage entry (one way of one set).
  // Total: 1 + 20 + 3 + 40 + 1 + 1 = 66b
  typedef struct packed {
    logic                     valid;
    logic [UBTB_TAG_BITS-1:0] tag;     // PC[26:7]
    logic [2:0]               br_type; // UBTB_BR_* encoding
    logic [VA_WIDTH-1:0]      target;  // predicted next fetch PC
    logic                     br_taken;// direction (COND only)
    logic                     carry;   // target in different

  } ubtb_entry_t;

  // Prediction output for one prediction slot.
  typedef struct packed {
    logic                     valid;
    logic [VA_WIDTH-1:0]      target;
    logic [2:0]               br_type;
    logic                     br_taken;
    logic                     carry;
  } ubtb_pred_t;

  // Update bundle for one prediction slot.
  // Driven post-execute by the resolution path.
  typedef struct packed {
    logic                     valid;
    logic [VA_WIDTH-1:0]      pc;
    logic [2:0]               br_type;
    logic [VA_WIDTH-1:0]      target;
    logic                     br_taken;
    logic                     carry;

#### Step 2 -- Write rtl/ubtb.sv

Write the complete ubtb.sv module from scratch.

Module signature:

  import bp_pkg::*;
  module ubtb #(
    parameter int NUM_PRED_SLOTS = 1
  )(
    input  logic                            clk,
    input  logic                            rstn,
    input  logic [VA_WIDTH-1:0]             pred_pc,
    output ubtb_pred_t [NUM_PRED_SLOTS-1:0] pred,
    input  ubtb_upd_t  [NUM_PRED_SLOTS-1:0] upd
  );

Index and tag derivation:

  Index: pred_pc[UBTB_IDX_BITS+1:2]
         PC[7:2], 6 bits, drops low 2 bits for 32b alignment.

  Tag:   pred_pc[UBTB_TAG_BITS+UBTB_IDX_BITS+1:UBTB_IDX_BITS+2]
         PC[26:7], 20 bits.

  carry: target[VA_WIDTH-1:5] != pc[VA_WIDTH-1:5]
         target is in a different 32B-aligned block than pc.

Storage:

  Declare the array as:
    ubtb_entry_t mem[UBTB_SETS][UBTB_WAYS];

  Reset: on rstn low, clear all valid bits only. Do not reset
  other fields (power saving).

  Replacement: per-set write pointer, width $clog2(UBTB_WAYS).
  Advances on each miss-write to that set.

Prediction path (combinational from registered mem):

  For each slot s in [0, NUM_PRED_SLOTS-1]:
    - Slot 0 uses pred_pc directly.
    - Slot 1 (when present) uses pred_pc + 32 (next 32B block).
    - Derive set index and tag for that slot's PC.
    - Search all UBTB_WAYS for a valid entry with matching tag.
    - Hit: drive pred[s] with entry fields, pred[s].valid=1.
    - Miss: drive pred[s].valid=0, all other fields 0.

Update path (synchronous):

  For each channel u in [0, NUM_PRED_SLOTS-1]:
    - When upd[u].valid:
      - Derive set index and tag from upd[u].pc.
      - Search all ways for a valid entry with matching tag.
      - Hit: update all fields in place.
      - Miss: write to the way at the current write pointer
        for that set. Advance the write pointer.

Read-during-write: prediction reads pre-update registered
state when upd and pred_pc index the same set in the same
cycle. No bypass. New entry visible on the following cycle.

Timing comment block at top of file:
  // uBTB pipeline timing:
  // s0: pred_pc presented. Index and tag derived comb.
  // s1: mem is registered. pred output is comb from mem.
  //     pred valid at start of s1 (one cycle after pred_pc).
  // Update: synchronous write on clk posedge when upd.valid.

### Verilator suppressions added

1. `-Wno-IMPORTSTAR` added to `VER_FLAGS` (affects all targets).
   **Justification:** The project mandates file-scope import
   (`import bp_pkg::*;` before module declaration). This style
   is correct per CLAUDE.md but Verilator 5.020 warns about
   wildcard imports in `$unit` scope because they can pollute
   the global namespace. Since the project uses a single
   package (bp_pkg) project-wide and the style is a fixed
   constraint, the warning is structural and must be suppressed
   permanently.

2. `-Wno-VARHIDDEN` added to `sim_ubtb` target only.
   **Justification:** `ubtb.sv` declares `parameter int
   NUM_PRED_SLOTS = 1` which shadows the identically named
   parameter in `bp_pkg`. This is intentional: the module
   parameter is the override point for instantiation; the
   package parameter is the default. Renaming the module
   parameter would violate the interface specification. The
   suppression is scoped to sim_ubtb only (not added to
   VER_FLAGS) because other modules may not share this pattern.

### Assumptions made not explicit in the prompt

1. **UBTB params partially pre-existing in bp_pkg.sv:** The
   file already contained `UBTB_ENTRIES`, `UBTB_WAYS`,
   `UBTB_SETS`, and `UBTB_IDX_BITS`. Step 1 added
   `UBTB_TAG_BITS` and the `UBTB_BR_*` localparams to the
   existing uBTB block rather than creating a duplicate block.

2. **TC10 pre-update check timing:** "verify pred[0] in cycle N
   reflects pre-update state" requires checking pred BEFORE
   the posedge, not after posedge+#1 as other TCs do. The
   combinational pred reads registered mem; checking between
   clock edges confirms no bypass. The initial TB structure
   placed the check after posedge and was corrected to check
   between edges.

3. **Dual reset in TC9/TC10:** rstn is shared between dut1
   and dut2. TC9 reset clears both DUTs. TC10 issues an
   additional reset for dut1 to ensure a clean state before
   the read-during-write test. This is safe given shared reset.

---

### RVA23 compliance risks and gaps noticed

1. **VA_WIDTH = 40b vs. Sv48:** RVA23 mandates Sv48 or Sv57
   support for supervisor mode (48b or 57b VA). The 40b
   parameter is consistent with Sv39+1 guard bit but does not
   cover Sv48. If Sv48 is required, `VA_WIDTH` must increase
   to 48 and the uBTB tag/index derivation updated accordingly.

2. **UBTB_BR_* vs. bp_br_type_e mismatch:** Two branch type
   encodings now coexist in bp_pkg: the existing `bp_br_type_e`
   enum (FTQ/update path) and the new `UBTB_BR_*` localparams
   (ubtb_entry_t). A translation step will be needed when the
   uBTB output drives bp_ftq_entry_t.br_type. Flagged for
   BP cluster integration (BP-005 or later).

3. **C-extension fetch block model:** The carry bit uses bit 5
   of the PC (32B block boundary). With C-extension, up to 16
   compressed instructions fit in 32B. The uBTB stores one
   target per fetch block; multiple taken branches per block
   are not modeled here (deferred to FTB at s2).
