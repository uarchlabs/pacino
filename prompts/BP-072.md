The open gap — external anchor

The verification is partly self-referential. The IA wrote the RTL geometry and the golden fold_ref to the same orientation it chose (increment-oriented walk, post-advance checkpoint, anchor ckpt−1). fold_ref is DUT-independent in the sense the prompt required (it reads ghr_buf, not the DUT's fold_step), so the requirement is met — but it is geometry-self-consistent, not externally anchored. If the chosen geometry were subtly wrong, both DUT paths would share the error, agree with each other and with fold_ref, and the suite would still be all-green.

incremental == recompute is necessary but not sufficient. The sufficiency check that's missing: does the new increment-oriented bp_history produce the same fold values that session-054 validated as Xiangshan-faithful for the TAGE/ITTAGE tables? handoff-054's whole premise was "tables are correct, only bp_history's bit order was wrong." The IA just changed bp_history's geometry again (direction + eviction position) to fit BP-069's pointer. Plausibly the fold values are preserved — same bits, same age order, different index bookkeeping — but the testbench never checks bp_history's output against an independent Xiangshan reference value or the table-consumption contract. It only checks it against the IA's own golden.

This is, incidentally, exactly the hole a hand-computed anchor vector would have closed — the thing I declined to add last turn because I didn't have the decisions-doc bit mapping. The impulse sweep I proposed proves fold_ref is well-formed (one-hot in → one-hot out at posmap), but it can't prove the chosen geometry is the design-correct one. So before Complete I'd want one external value: pick a single (H,W) and a known history, compute the expected fold from the table hash path or a reference Xiangshan computation, and confirm the new DUT output equals it.

Two decisions that are yours to ratify, not the suite's

The IA didn't just land RTL — it made two architectural calls: (1) increment-oriented fold geometry, and (2) post-advance checkpoint with anchor ckpt−1. Under dictate-vs-propose, pointer and checkpoint architecture are yours to dictate; the IA proposed and verified, which is fine, but green ≠ ratified. It also edited bp_history_decisions.md to record its resolution — i.e., an agent rewrote an authority doc to match its own RTL choice, and in doing so overrode this prompt's own Binding Decision ("checkpoint captures the pre-advance pointer, decisions §6"). Offline equivalence that doesn't exercise the integration is where these hide.

## Decisions made not explicit in the prompt
- SCOPE EXPANSION (user authorized): produced and corrected
  bp_history.sv rather than treating it as read-only. The prompt's
  premise that BP-069 had landed in rtl/ was false; the BP-069 RTL
  lived only in versions/bp_history.sv and still carried the
  pre-BP-071 (32b, pos=i%W) fold geometry, while the active rtl/ file
  carried BP-071 folds on the old caller-owned interface. Neither was
  the intended DUT.
- FOLD GEOMETRY (increment-oriented). BP-071's geometry assumes a
  decrementing pointer (newest at ptr, older at ptr+i, leaving bit at
  ptr+H). With BP-069's incrementing pointer this makes the
  incremental fold diverge from the rollback recompute (empirically:
  rolling back to an un-diverged checkpoint changed folded
  0x92 -> 0x00). Fix: fold_ghr walks DOWNWARD (ptr-i); fold_step
  evicts the leaving bit at (write_addr - H). The BP-071 position
  mapping posmap(i)=(i+W-1)%W and high-end insertion are unchanged.
- CHECKPOINT ANCHOR (post-advance). The checkpoint now stores the
  POST-advance live pointer; rollback restores it and recomputes
  folds from anchor = ckpt - 1 (the newest bit of the bundle). This
  is required for dual-slot rollback to match the incremental fold,
  and it aligns the RTL with bp_history_interfaces.md (which already
  specified post-advance) and resolves the contradiction with
  decisions.md section 6 in favor of the interface doc.
- GOLDEN ANCHOR. The single reference is anchored at ghist_ptr-1 and
  walks downward, so one fold_ref matches both the incremental and
  the recomputed DUT outputs in every settled cycle.
- The two-if slot structure (decisions 3.5 if/else-if cleanup) was
  left as-is (functionally correct nonblocking last-write-wins, as in
  both prior files) to minimize deviation from the reviewed logic;
  the cleanup remains deferred.
