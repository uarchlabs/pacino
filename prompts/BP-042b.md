## Hypothesis
Two minimal fixes. ittage_assert.sv assertion 2 is missing

sim_tage_manual and causes a double-bind. Both are one

## Background

DEFERRED-3: ittage_assert.sv assertion 2 checks
using_primary=1 implies prm_comp>0. This is only valid
when a tagged table hit occurred (ittage_hit=1). When
ittage_hit=0 the RTL legitimately has prm_comp=0 and
using_primary=1 as a default state -- no provider was
selected so using_primary is a don't-care. The assertion
fires incorrectly during TC-P01 no-hit predictions.

Assertions 1 and 3 in ittage_assert.sv already gate on
ittage_hit. Assertion 2 is missing this guard. The fix
adds ittage_hit to the condition on both the pred path
and the update path within assertion 2's always_ff block.

D2: BP-042a added an inline bind for tage_assert inside
tb_tage_manual.sv. The sim_tage_manual Makefile target
still compiles tage_assert_bind.sv, which also contains
a bind for tage_assert. This results in two tage_assert
instances bound to tage -- one from the inline bind in
tb_tage_manual.sv and one from tage_assert_bind.sv.
Remove tage_assert_bind.sv from the sim_tage_manual
Makefile target to eliminate the double-bind.
