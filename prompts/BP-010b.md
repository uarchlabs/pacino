## Background

Read tage_tb_decisions.md in full before proceeding.
It is the authoritative source for all design decisions.

BP-010a delivered working RTL and a passing tage_rdy_tst
under both sim targets. However the fast init path is
functionally incorrect: when TAGE_FAST_INIT=1 is passed,
sram_init still sequences all entries and tage_rdy does
not assert until ~2049 cycles. The intent is that
TAGE_FAST_INIT=1 fully bypasses sram_init and tage_rdy
asserts on the first cycle after reset deasserts.

Three fixes are required:

Fix 1 -- tage.sv: full sram_init bypass when fast_init=1.
tage.sv must read +TAGE_FAST_INIT into a logic register
fast_init_r via an initial block. All tbl_ri_* signals
driven to the table instances must be muxed on fast_init_r:
when fast_init_r=1 they are all strapped to zero and
tage_rdy is driven 1'b1 directly. When fast_init_r=0
they pass through from sram_init as before and tage_rdy
follows tbl_ri_rdy. sram_init is still instantiated
but its outputs are ignored when fast_init_r=1.

Fix 2 -- tage_bim.sv and tage_table.sv: 2D mem access.
Read bw_ram.sv to confirm the mem array declaration.
The mem array is 2D: mem[BANKS][ENTRIES]. The initial
blocks added in BP-010a use 1D access mem[i] which is
incorrect. Fix to use nested loops: outer over banks
(b=0..1), inner over entries (i=0..RAM_ENTRIES-1),
accessing mem[b][i]. Apply to both u_ram_s0 and
u_ram_s1 in both modules.

Fix 3 -- tb_tage.sv: cycle count window.
The TAGE_FAST_INIT=0 cycle count check must verify
elapsed cycles is less than (1 << MAX_IDX_WIDTH) + 4.
The current check uses a range that may be too tight.
When TAGE_FAST_INIT=1, tage_rdy asserts on the first
cycle after reset deasserts. No cycle count check is
performed in this path.
