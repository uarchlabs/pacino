# Task Overview

TD #66 structural rework. The TAGE update and allocation buses
between tage_cntrl and the per-table instances are currently
declared as 2D arrays dimensioned [table][slot]. Only one table
per slot is the provider / allocation target at update time, so
the per-table dimension carries TAGE_NUM_TABLES-1 unused slices
on every update. The correct structure is a single shared bus
per slot, with the target table identified by the corresponding
*_tbl_sel_u0 select that already exists. This is a pure refactor:
no behavioral change is intended and the existing 68-test
tb_tage suite is the regression gate.

## Hypothesis

Collapsing the per-table 2D update/allocation buses to a single
shared bus per prediction slot, with the target table carried by
the existing *_tbl_sel_u0 selects, is behavior preserving. After
the change, lint is clean and all 68 tb_tage tests pass with no
expected-value edits, confirming the rework is purely structural.

## Background

tage_cntrl currently drives TAGE_NUM_TABLES copies of each
update and allocation bus, one slice per table, dimensioned
[0:TAGE_NUM_TABLES-1][0:NUM_PRED_SLOTS-1]. At update time only
one table per slot is the provider (or the allocation target),
identified by the matching *_tbl_sel_u0 select. The per-table
dimension is therefore redundant: every update drives one
meaningful slice and TAGE_NUM_TABLES-1 unused ones.

The correct form keeps only the bus width and the prediction
slot as dimensions:

    output logic [TAGE_MAX_CTR_WIDTH-1:0]
      t_prm_ctr_wd_u0[0:TAGE_NUM_TABLES-1][0:NUM_PRED_SLOTS-1]

becomes

    output logic [TAGE_MAX_CTR_WIDTH-1:0]
      t_prm_ctr_wd_u0[0:NUM_PRED_SLOTS-1]

The bus is shared across all table instances; the consuming
table identifies whether the transaction applies to it from the
corresponding *_tbl_sel_u0 value, gated by the write-enable and
match logic that already exists in tage_table.

## Assumptions made not explicit in the prompt

The prompt states "only one table per slot is the provider (or the
allocation target)." This holds for allocation and for USE/EPC writes.
It does NOT hold for CTR writes: prm CTR and alt CTR can both fire
in the same cycle to different tables with different addresses.
This is not an incorrect assumption in the prompt -- the use of
"provider or allocation target" intentionally excludes the alt CTR
case. The refactor handles it correctly via t_alt_upd_index_u0.

## Decisions made not explicit in the prompt

t_upd_index_u0 cannot be a single shared [slot] bus while
preserving behavior. Analysis: when both u_prm_ctr_wr and
u_alt_ctr_wr are asserted in the same cycle (legal per the
CTR update rules), prm table T_p needs u_prm_idx and alt table
T_a needs u_alt_idx. These are in general different values.
A single shared bus can carry only one value per slot.

Decision: add t_alt_upd_index_u0[0:NUM_PRED_SLOTS-1] as a
companion output of tage_cntrl. It carries u_alt_idx[s].
t_upd_index_u0[s] carries u_prm_idx[s]. In tage.sv gen_tage_tbl,
the per-instance upd_idx_w is computed with:
  (TBL_SEL_W'(t) == w_prm_tbl_sel_u0[s])
    ? w_upd_index_u0[s]   -- prm index for prm table
    : w_alt_upd_index_u0[s]  -- alt index for alt table
This is the "index-vs-select comparison at the tage.sv connection
boundary" permitted by requirement 5. tage_table.sv receives the
correct pre-selected index via upd_idx_w and needs no changes.
