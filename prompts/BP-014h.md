### Aging mechanism summary

From tage_cntrl_uaon_update_rules.md:

u_eff is computed at predict time from the raw USEFUL
field and the difference between the current epoch
counter and the entry's EPC field:

  age = (lcl_epoch - EPC) mod 4

  if (age == 0) u_eff = USEFUL
  if (age == 1) u_eff = USEFUL >> 1
  else          u_eff = 0

lcl_epoch is a 2b wrapping counter in tage_cntrl.sv,
one per slot (lcl_epoch_0, lcl_epoch_1).
EPC is the 2b epoch field stored in each T1-T4 entry.

With aging disabled (tage_enable_aging=0) and
lcl_epoch=0, EPC=0: age=0, u_eff=USEFUL always.
These tests require lcl_epoch != EPC to exercise
the aging formula. lcl_epoch is advanced via
hierarchical write before predicting.

On a USE write at update time, lcl_epoch is written
to the EPC field of the updated entry alongside the
new USEFUL value.

### Aging effect on allocation candidate selection

From tage_cntrl_alloc_rules.md:
  Candidate selection scans for u_eff == 0.
  u_eff is used for the useful == 0 test, not raw USEFUL.

Therefore an entry with raw USEFUL=2'b10 and age=2
has u_eff=0 and IS a valid allocation candidate,
even though USEFUL != 0.

An entry with raw USEFUL=2'b10 and age=1 has
u_eff=2'b10>>1=2'b01, which is != 0 and is NOT
a valid allocation candidate.

   fields are written independently via separate

   bit ranges: CTR=[3:1], USE=[5:4], EPC=[7:6].
