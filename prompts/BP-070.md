bp_history_decisions.md and bp_history_interfaces.md are the

## Summary
Status: ABANDONED due to a specification error (please set the Status
header box to abandoned). The error is in Specific Requirement 3
(TD #74) and bp_history_decisions.md section 3.3.

The spec asserts the incremental dual-slot fold equals a forward
fold_ghr linear walk from the bundle-start pointer. Requirement 3(b)
reads: "recompute reference: walk the post-update buffer linearly from
the bundle-start pointer", and decisions 3.3 calls this "fold_ghr
order". The as-built RTL (frozen, read-only here) does not behave that
way. The incremental fold is RECENCY-ORDERED: the newest written bit
lands at fold position 0, which is the BIT-REVERSE of the fold_ghr
forward walk. Forward fold_ghr matches only the ROLLBACK recompute
path, not the incremental path. Therefore the (a)==(b) equivalence as
literally written is false against the RTL, and a TD #74 test cannot be
both faithful to the spec's literal (b) and green.

Evidence (directed probe, TAGE T1 with H=8, W=8, no fold wrap):
- Single-step bits written at GHR positions 0..7 = 0100_1101
  (ghr_buf[7:0]); pos 0 is the first/oldest written.
- DUT folded.tage_t1_idx_fh[7:0] = 1011_0010.
- Forward fold_ghr(ghr_buf, start=0, H=8, W=8) = 0100_1101.
- 1011_0010 is the exact bit-reverse of 0100_1101. The DUT incremental
  fold is recency order; the spec's forward fold_ghr is the reverse.
- Dual-slot fresh case confirms it: slot0=0 at ptr0, slot1=1 at ptr1
  gives DUT idx fold bit0=1 (newest at position 0), forward walk puts
  it at bit1.

A working testbench was developed during investigation and
sim_history passed 18/18 cases (12896 assertions), but ONLY by using a
reference that matches the RTL's actual behavior (recency order for the
incremental path, forward fold_ghr for the rollback path). That is a
deviation from the spec's literal 3(b) definition. Rather than paper
over the contradiction, the task is abandoned for spec correction. Jeff
will revert the testbench.

Recommended spec fix (one of):
- Correct 3(b) / decisions 3.3 to state the incremental fold is recency
  ordered (newest bit at fold position 0), and define the independent
  recompute reference as a recency (newest-first) walk, not a forward
  fold_ghr walk. Keep fold_ghr (forward) only for the rollback
  recompute reference (requirement 5). The achievable, true equivalence
  is: incremental two-step == two sequential single-slot cycles ==
  recency recompute == DUT; OR
- Change the RTL so the incremental fold and the rollback recompute use
  the same ordering (currently they disagree -- see RVA23 risks below),
  then re-issue the testbench task.

## RVA23 compliance risks and gaps noticed
- SPEC ERROR (cause of abandonment): requirement 3(b) and
  decisions 3.3 specify a forward fold_ghr walk for the incremental
  dual-slot recompute reference. The RTL incremental fold is recency
  ordered (bit-reverse of fold_ghr). The two cannot both be true; the
  spec must be corrected (or the RTL changed).
- RTL INCONSISTENCY (design risk, flag for review): the incremental
  update path and the rollback recompute path use DIFFERENT fold bit
  orderings. Incremental fold_step builds recency order (newest at
  position 0); the rollback branch recomputes with fold_ghr (forward
  order). A fold rebuilt by rollback therefore differs in bit-order
  from the fold the incremental path would have held at the same
  pointer. Because the tables are advisory this does not corrupt
  architectural state, but predictor accuracy across a rollback is
  affected and the "history is restored" intent is not literally met.
- ROLLBACK RECOMPUTE WINDOW (design risk): fold_ghr walks FORWARD from
  the restored pointer (positions ptr..ptr+H-1), i.e. buffer entries
  written AFTER the checkpoint (the contamination region per
  decisions 6), not the H bits of history preceding the pointer. The
  rollback fold thus folds post-checkpoint/forward bits rather than
  the pre-checkpoint history. Confirm this is intended.
- fold_step uses a 32-bit accumulator while SC ST3 has W=64
  (SC_MAX_FH=64). The ST3 idx fold is effectively computed in 32 bits
  (upper bits zero, W-wrap term disabled). Not the abandonment cause,
  but flagged: the SC ST3 folded width and the fold datapath width
  disagree.
