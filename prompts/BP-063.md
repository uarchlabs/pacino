## Background
ras.sv was implemented in BP-062. It is a register-file-based
dual-stack RAS with no SRAMs. The module under test has these
primary behaviors to verify:

  p0 TOS read: combinational, reflects current TOSR.
  p2 push: DIRECT_CALL or INDIRECT_CALL, slot 0 before slot 1.
  p2 pop: RETURN, produces ras_pop_addr_p2.
  Same-cycle bypass: slot0=call + slot1=return in same cycle.
  Recursion counter: increment on matching push, decrement on pop.
  Snapshot output: post-op TOSR/TOSW/BOS per slot at p2.
  Mispredict restore: pointer-only, blocks p2 push/pop.
  p3 repair: inverse op when p3 br_type disagrees with p2.
  Commit stack: push on call commit, pop on return commit.
  Empty fallback: speculative empty -> serve from commit stack.
  Overflow: silent wrap, no fault.

Note: planning documents use s-stage notation (s0/s1/s2/s3).
RTL port names use p-stage notation (p0/p1/p2/p3). They are
equivalent. Use p-stage in all RTL and testbench.

## Binding Previous Decisions
- Slot 0 processed before slot 1 in all dual-slot cases.
  (ras_decisions.md section 6.2, ras_interfaces.md IC-RAS-03)
- Same-cycle bypass: slot0=call + slot1=return forwards slot0
  push address directly to slot1 pop output without array read.
  (ras_interfaces.md IC-RAS-04)
- Recursion counter: push increments rctr when push address
  matches current TOS ret_addr; pop decrements rctr before
  moving TOSR. (ras_decisions.md section 5, IC-RAS-05)
- ras_restore_val blocks p2 push/pop in the same cycle.
  (ras_interfaces.md IC-RAS-09)
- p3 repair table (ras_decisions.md section 1):
    p2=push, p3=no-op  -> repair: pop
    p2=no-op, p3=pop   -> repair: pop
    p2=pop,  p3=no-op  -> repair: push
    p2=no-op, p3=push  -> repair: push
- Empty fallback: when speculative stack empty (TOSR==BOS),
  serve pop from commit stack top. (ras_interfaces.md IC-RAS-07)
- Overflow: silent circular wrap, no error signal.
  (ras_interfaces.md IC-RAS-06)
- One commit event per cycle. (ras_interfaces.md IC-RAS-10)

## Defect Report (ras.sv)

DEFECT 1 -- Off-by-one in speculative-stack empty/valid (FIXED).

Original mechanism:
  - Reset: tosr = tosw = bos = 0.
  - A normal push wrote at tosw, set tosr = tosw(old), advanced tosw.
    After one push from reset: tosr = 0, tosw = 1, bos = 0.
  - Validity/emptiness is tested as (tosr != bos). After one push
    tosr == bos == 0, so the stack read EMPTY with one live entry.
  Consequences (confirmed pre-fix, 10 PASS / 6 FAIL run): single
  entry invisible at p0; a later-cycle pop fell through to the commit
  stack; the bottom-most entry could never be popped.

Fix (re-base allocation; option chosen by Jeff):
  - The empty test (TOSR==BOS) is semantically correct ("speculative
    stack drained to the committed floor"); the bug was that the
    first allocation landed on the BOS slot.
  - Both allocation sites (p2 normal push, p3 repair re-push) now
    compute the write index as:
        w_alloc = (w_tosw == bos) ? (w_tosw + 1) : w_tosw;
    so allocation skips the BOS sentinel only when TOSW would land on
    it (cold-start, or a full circular wrap). TOSW remains monotonic
    across pops, preserving the pointer-only mispredict-restore
    guarantee (ras_decisions.md 3.2 / 4.3).
  - Cost: usable speculative depth is 15 (one sentinel slot), not 16.
    Flagged for the decision doc.
  - Reset values are unchanged (all 0).

Note: ras_decisions.md 3.2 prose ("Empty condition: TOSR == BOS" with
"Push: TOSR = TOSW. TOSW advances") still reads as the original
off-by-one and should be reconciled to state the BOS-sentinel /
allocation-above-BOS rule and the 15-entry usable depth.

DEFECT 2 -- p3 undo-pop repair allocates instead of re-exposing
(FIXED).

Location: ras.sv rep_push branch in the p3 repair pass.

Mechanism:
  - rep_push covers two IC-RAS-11 cases with identical logic:
      (a) s2=no-op, s3=push  -> apply a missed push  (correct to
          allocate a new entry from the registered fallthrough), and
      (b) s2=pop,  s3=no-op  -> undo an erroneous pop.
  - For case (b), undoing a pop should re-expose the entry the pop
    consumed (it is still resident; pop does not overwrite). That is
    a TOSR increment back to the pre-pop top. Instead, rep_push
    allocates at the TOSW frontier and writes p3_fallthr_q (the
    fallthrough sampled during the pop cycle, which is not a call
    return address), advancing TOSR/TOSW.

Observed (TC-17): seed one entry X at idx1 (tosr=1, tosw=2), pop
(tosr=0), then p3 disagrees (no-op vs the pop):
  expected pre-pop state tosr=1, top=X (0x9000)
  observed              tosr=2, tosw=3, top=spec[2]=0x0000
The pointer did not return to pre-pop and the re-exposed top is a
stale/zero entry, not X.

Fix applied (authorized): the rep_push path is split. When
p3_op_q==OP_POP (undo a pop), the repair performs a TOSR-only
re-expose: w_tosr is incremented back to the popped slot and
w_top_addr/w_top_rctr are reloaded from the array; no array write,
TOSW unchanged. When p3_op_q==OP_NONE & s3=call (missed push), the
original frontier allocation of the registered fallthrough is kept.
After the fix TC-17 observes tosr=1, top=X (0x9000), as expected.

Limitation (documented): the undo-pop re-expose reverses a normal
(TOSR-moving) pop. Reversing a recursion pop (one that decremented an
rctr without moving TOSR) is not separately handled, because the
pre-pop rctr is not recoverable from post-pop working state; this
mirrors the inherent asymmetry already present in rep_pop and is the
TOSR-only behavior Jeff selected. Not exercised by an s2/s3 repair in
the directed suite.

Let me work the invariant before recommending, because this is exactly where I failed before.
The relevant fact, stated in ras_decisions.md §3.2 and confirmed by Claude Code: pop does not overwrite data. TOSR decrements, the entry stays resident in the array. That is the whole basis of pointer-only restore.
So an erroneous pop (s2=pop) consumed an entry that is still physically in the array. The p3 repair for that case is "undo the pop." Undoing a pop that only moved TOSR means: move TOSR back. The entry is still there; you re-expose it by incrementing TOSR and reloading top/rctr from the array. You do not allocate a new frontier slot, and you do not write the registered fallthrough — there is no new entry to create, and the fallthrough address is meaningless in this case because no call happened.
Claude Code's diagnosis is correct and its proposed fix direction is correct. The current code wrongly routes undo-pop through the same allocate-and-write path as missed-push. Those two repair cases are not the same operation:

missed-push (s2=no-op, s3=push): a call should have happened and didn't. Allocate at frontier, write the fallthrough. New entry is correct.
undo-pop (s2=pop, s3=no-op): a pop happened that shouldn't have. The entry is still resident. Re-expose it by moving TOSR back. No allocation, no write.

This mirrors how rep_pop already splits its own sub-cases, which tells you the asymmetry is expected in this design, not a special case being bolted on.
Recommendation: Option 1. Split rep_push so undo-pop does TOSR-only re-expose (increment TOSR, reload top addr and rctr from the resident array entry), while missed-push keeps frontier allocation.
