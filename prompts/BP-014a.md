### Dual tagged provider setup

requires:
  - T1 entry at idx_hash(PC): valid=1, tag matching PC
  - T2 entry at idx_hash(PC): valid=1, tag matching PC

Index hash formula (from tage_table_interfaces.md):
  idx_hash = (pc >> INST_OFFSET) ^ fh_idx
  truncated to THIS_INDEX_BITS (11b).
  With folded_hist=0: idx_hash = (pc >> 2)[10:0]

Tag hash formula:
  tag_hash = (pc >> THIS_INDEX_BITS) ^ fh1 ^ (fh2<<1)
  truncated to THIS_TAG_BITS (8b).
  With folded_hist=0: tag_hash = (pc >> 11)[7:0]

T1 and T2 use the same index and tag hash formulas
with folded_hist=0, so for a given PC both tables
index and tag the same way. Choose a PC such that:
  - idx_hash maps to a convenient bank/row address
  - tag_hash is nonzero (avoid ambiguity with zero-init)
  - Address does not overlap any prior test address

Bank address assignment (from tage_table_interfaces.md):
  bank_addr = index[THIS_INDEX_BITS-1]  (bit 10)
  row_addr  = index[THIS_INDEX_BITS-2:0] (bits 9:0)

## RVA23 compliance risks and gaps noticed

R1. Provider labeling inconsistency in planning documents:
    The prompt and some planning documents refer to T1 as
    primary and T2 as alternate in the four-test scenario.
    RTL ascending scan makes T2 primary when both T1 and
    T2 hit. This is not an RTL defect (behavior matches
    TAGE algorithm), but the documentation is misleading.
    No RVA23 violation; TAGE is a microarchitectural
    optimization not subject to RVA23 specification.

R2. UAON counter frozen by pred_strong gate: when the
    post-mux prediction is strong (CTR not in {011,100}),
    uaon_upd_ff does not update. In TC-31, T1 CTR=101
    produces pred_strong=1, freezing uaon at 4'h8. The
    UAON counter would only decrease if the prm/alt agree
    with prediction direction when prm is correct. This
    edge case may cause the UAON counter to drift high
    and remain there, permanently using the alternate
    provider even if the primary improves. Worth reviewing
    the threshold and update policy in future sessions.
